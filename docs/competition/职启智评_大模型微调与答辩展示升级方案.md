# 职启智评大模型微调与答辩展示升级方案

更新时间：2026-05-16
适用项目：`D:\apps\ai-interview`
用途：比赛答辩、后续升级路线、评委问答准备、功能演示包装
当前原则：不伪造已完成训练，不宣称自研底层大模型；把已经实现的“岗位画像 + RAG + 授权样本 + 人工评分 + 评测样本导出”组织成可展示的微调数据闭环。

## 1. 答辩定位

### 1.1 一句话主张

职启智评不是简单调用通用 ChatGPT 问答，而是构建了面向就业能力诊断的“岗位画像/RAG/证据评分/人工标注/微调样本沉淀”闭环，为后续面向求职场景的大模型微调与持续优化提供可落地的数据基础。

### 1.2 对评委的核心表达

我们当前重点不是炫耀“训练了一个大模型”，而是解决大模型落地最难的三件事：

1. 什么样的数据可以进入训练和评测。
2. 如何保证简历、面试、报告和人工评分之间证据一致。
3. 如何把用户授权后的真实案例沉淀为可持续优化的样本。

这比单纯接一个大模型 API 更贴近真实产品，因为模型能力提升最终依赖高质量场景数据、人工评价标准和可追溯闭环。

### 1.3 不能说和可以说

| 类型 | 不建议说 | 建议说 |
|---|---|---|
| 底层模型 | 已经训练出自研大模型 | 基于通用大模型构建就业能力诊断应用，并设计了可用于后续微调的数据闭环 |
| 微调状态 | 已完成大规模微调 | 已完成微调数据准备链路和人工评分沉淀入口，后续可基于授权样本进行 SFT/偏好优化 |
| 实验结论 | 微调后效果显著提升 | 目前先完成闭环和样本标准，真实 C1/C2/C3 跑测后再给出量化结论 |
| 数据规模 | 已有大量真实训练数据 | 已建立授权采集、去标识化、人工标注和评测样本导出流程 |
| 创新点 | 我们比通用模型更聪明 | 我们把通用模型接入岗位画像、证据约束和人工评分闭环，使其更适合就业训练场景 |

## 2. 当前项目状态自适应结论

### 2.1 已经能支撑答辩的真实能力

| 能力 | 当前项目状态 | 答辩说法 |
|---|---|---|
| 简历解析 | 已支持 PDF 文本抽取、可选 OCR、`parse_quality`、`normalized_resume` | 不是简单上传文件，而是对简历质量和结构化结果做诊断 |
| 岗位画像 | 已有标准岗位画像、成员资料归并路线、知识切片 | 用岗位画像约束大模型，不让大模型自由发挥 |
| RAG 追问 | 面试路由可携带岗位切片、验证目标和证据状态 | 面试问题来自岗位要求和能力缺口，而不是随机聊天 |
| 证据评分 | `resume_evaluation_snapshot` 统一简历评分、润色、学习计划、面试和复盘口径 | 评分有证据状态，区分已证明、间接证明、仅声明和缺失 |
| 学习任务 | 已有服务器持久化和阶段 139 质量提示 | 诊断不是结束，而是能转成学习任务 |
| 人工标注 | 后台案例工作台和面试详情页支持人工评分沉淀，未授权不能保存 | 训练样本不是自动生成后直接用，而是需要人工质量控制 |
| 数据授权 | 注册、上传、报告、复盘、个人设置已有基础协议和可选数据贡献授权 | 真实样本进入评测和微调准备前有授权留痕 |
| 评测样本 | 后台评测样本页展示“已授权完成测评”，后端导出只取授权样本 | 已经有面向微调/评测的数据出口雏形 |

### 2.2 还不能夸大的部分

| 项目 | 当前真实状态 | 答辩处理 |
|---|---|---|
| C1/C2/C3 真实闭环 | 仍待服务器跑测和 CSV 回填 | 说“正在验收”，不说“已通过” |
| 大规模真实数据 | 尚未形成足量真实样本 | 说“已完成采集和标注机制”，不说“已有大规模数据集” |
| 模型微调 | 尚未完成实际 SFT/LoRA 训练和评测 | 说“已具备微调数据准备与后续训练路线” |
| 人工评分一致性 | 未形成双评分统计 | 说“后台已支持评分沉淀”，不说“已证明一致性高” |
| 效果提升百分比 | 还没有真实对照实验 | 不写具体提升数字 |

## 3. 微调叙事框架

### 3.1 三层模型增强

把项目讲成三层，不要只讲“调用大模型”。

| 层级 | 名称 | 作用 | 当前状态 |
|---|---|---|---|
| 第一层 | 提示词与结构化输出约束 | 让模型按简历证据、岗位能力、报告结构输出 | 已实现 |
| 第二层 | RAG 与岗位知识切片 | 让模型围绕岗位要求、面经、能力模型追问 | 已实现初版 |
| 第三层 | 授权样本与人工评分驱动的微调准备 | 把真实案例转为训练/评测样本，后续做 SFT/偏好优化 | 数据链路已建，训练待后续 |

答辩时可以说：

> 我们不是直接把简历丢给大模型，而是先用岗位画像和证据状态约束它，再通过后台人工评分把真实案例沉淀为后续微调样本。这个闭环让系统具备持续进化能力。

### 3.2 数据飞轮

推荐在 PPT 中画成闭环图：

`用户授权 -> 简历解析 -> 岗位画像匹配 -> RAG 面试追问 -> 报告与学习任务 -> 后台人工评分 -> 去标识化训练样本 -> 微调/提示词/RAG 优化 -> 下一轮更准`

每一段都能对应项目现有模块：

| 闭环节点 | 对应项目模块 |
---|---|
| 用户授权 | 隐私协议、数据贡献授权、报告/复盘授权卡 |
| 简历解析 | ResumeUpload、resume_text_extractor、resume_normalizer |
| 岗位画像匹配 | position_knowledge_bases、岗位知识切片 |
| RAG 面试追问 | interview_service 路由和验证目标 |
| 报告与学习任务 | Report、TrainingReview、LearningTasks |
| 后台人工评分 | Cases、InterviewDetail、training_sample_review |
| 去标识化训练样本 | EvaluationDatasets、training sample export |
| 微调/优化 | 下一阶段 SFT/LoRA/偏好优化路线 |

## 4. 答辩展示内容设计

### 4.1 推荐新增 PPT/演示页

| 页面 | 标题 | 展示重点 |
|---|---|---|
| P1 | 为什么不是普通 AI 面试官 | 展示“通用问答”和“岗位证据闭环”的区别 |
| P2 | 大模型能力增强架构 | 展示 Prompt 约束、RAG、人工评分、微调准备三层 |
| P3 | 授权样本数据飞轮 | 展示从用户授权到训练样本沉淀的闭环 |
| P4 | 人工评分与微调样本池 | 展示后台案例标注、授权准入、评测样本规则 |
| P5 | 后续微调路线 | 展示 SFT、LoRA/QLoRA、偏好优化、A/B 对照实验 |

### 4.2 演示顺序

1. 用户上传简历，展示解析质量和结构化简历。
2. 展示能力诊断，强调证据状态不是模型自由判断。
3. 加入学习任务，展示任务质量标签。
4. 开始面试，展示问题围绕能力缺口追问。
5. 报告页展示“本次案例数据贡献授权”。
6. 后台案例工作台展示授权状态、人工评分和导出准入。
7. 评测样本页展示 `data_contribution_consent=true` 和人工标注规则。
8. 解释这些授权高质量样本将作为后续微调数据。

### 4.3 评委容易记住的表达

- 通用大模型给的是答案，我们做的是就业能力诊断闭环。
- 不是把岗位知识写进候选人经历，而是用岗位知识约束追问和评价。
- 不是所有数据都进入训练，只有用户授权、去标识化、后台人工评分后的样本才进入沉淀。
- 微调不是一句口号，前提是有可追溯的数据、评分规则和质量筛选。
- 我们当前已经把微调前最难的数据闭环、授权闭环和评分闭环做出来。

## 5. 后续功能升级路线

### 5.1 阶段 141：微调样本导出与答辩展示增强

目标：先不训练模型，先把“可用于微调的数据形态”做出来，方便答辩展示。

建议功能：

1. 后台评测样本页新增“微调样本预览”区域。
2. 导出 JSONL 格式训练样本：
   - `instruction`：岗位、能力缺口、面试任务。
   - `input`：去标识化简历摘要、岗位画像切片、面试上下文。
   - `output`：高质量追问、评分理由、报告建议或学习任务建议。
   - `metadata`：岗位、能力项、证据状态、人工评分、是否高质量、是否有幻觉。
3. 导出前继续要求：
   - `data_contribution_consent=true`
   - `training_sample_review.review_status=reviewed`
   - `has_hallucination=false` 或单独进入反例集
4. 增加样本集统计：
   - 已授权样本数
   - 已人工评分样本数
   - 高质量样本数
   - 幻觉反例数
   - 可用于 SFT 的样本数

答辩价值：评委可以看到不是空说微调，而是已经能把业务数据转成训练样本格式。

### 5.2 阶段 142：微调准备报告自动生成

目标：生成一份“模型优化准备报告”，用于答辩或申报材料。

报告字段：

| 字段 | 内容 |
|---|---|
| 数据来源 | 授权简历、授权面试、后台人工评分 |
| 去标识化策略 | 删除姓名、手机号、邮箱、证件号、学号、详细住址和文件名标识 |
| 保留字段 | 学校、专业、项目经历、技能、目标岗位、面试问答、评分信息 |
| 样本分层 | 高质量正例、幻觉反例、追问优质样本、报告建议优质样本 |
| 可训练任务 | 追问生成、评分理由生成、报告建议生成、学习任务生成 |
| 风险控制 | 未授权样本不导出；岗位知识不写成候选人经历 |

### 5.3 阶段 143：轻量微调实验

目标：在数据量足够后，真正做一轮轻量实验。

建议不要一上来追求大模型全量训练，而是选择低成本路线：

1. 先使用 30-50 条授权高质量样本做格式验证。
2. 使用 100-300 条样本后再做初步 SFT。
3. 如果硬件不足，优先做：
   - LoRA/QLoRA 小规模实验。
   - 或使用平台 API 的微调能力。
   - 或先做提示词优化/RAG 排序优化作为对照。
4. 对照组：
   - 通用大模型直接回答。
   - RAG + 提示词约束。
   - RAG + 人工评分样本优化。
   - 微调模型。
5. 指标：
   - 证据对齐率。
   - 幻觉率。
   - 追问质量。
   - 报告可执行性。
   - 学习任务可执行性。
   - AI 与人工评分差异。

### 5.4 阶段 144：偏好优化与反例集

目标：把人工评分不只是当分数，而是变成偏好数据。

样本类型：

| 样本类型 | 用途 |
|---|---|
| 高质量追问 | 学习什么问题更能验证能力 |
| 幻觉反例 | 学习不能把岗位知识写成候选人经历 |
| 报告建议优质样本 | 学习什么建议更可执行 |
| 学习任务优质样本 | 学习如何生成有材料、练习、验收方式、耗时的任务 |
| AI 与人工分歧样本 | 定位评分偏高或偏低的问题 |

## 6. 可以立刻落地的 MD/PPT 素材

### 6.1 技术架构图文案

标题：大模型驱动的就业能力诊断闭环

内容：

`简历解析层 -> 岗位画像/RAG 层 -> 证据评分层 -> 面试追问层 -> 报告与学习任务层 -> 人工评分层 -> 授权样本与微调准备层`

强调：

- 大模型负责理解、生成、追问和总结。
- 岗位画像和 RAG 负责约束方向。
- 证据快照负责统一口径。
- 人工评分负责质量控制。
- 授权样本负责后续微调和持续优化。

### 6.2 微调数据样本结构示例

```json
{
  "instruction": "你是就业能力诊断平台的面试追问模型，请根据岗位画像和候选人简历证据，生成一个能验证能力缺口的追问。",
  "input": {
    "target_position": "Python后端开发工程师",
    "ability_gap": "缓存与 Redis 应用",
    "evidence_status": "claimed_only",
    "resume_summary": "候选人声明熟悉 Redis，但项目经历中缺少缓存一致性或故障排查证据。",
    "rag_slices": [
      "岗位要求：理解 Redis 缓存穿透、击穿、雪崩和一致性处理。",
      "面试经验：追问候选人如何处理缓存与数据库双写一致性。"
    ]
  },
  "output": {
    "question": "你简历中提到熟悉 Redis。请结合一个具体场景说明，如果缓存和数据库出现短时间不一致，你会如何定位问题并设计修复方案？",
    "verification_target": "验证候选人是否真正理解缓存一致性、故障定位和工程取舍。"
  },
  "metadata": {
    "data_contribution_consent": true,
    "review_status": "reviewed",
    "human_overall_score": 8.5,
    "has_hallucination": false,
    "quality_tier": "high"
  }
}
```

说明：这只是答辩展示用结构样例；真实样本必须来自用户授权、去标识化和后台人工评分。

### 6.3 答辩口播模板

> 我们没有把项目停留在“接入一个大模型 API”。职启智评把岗位画像、简历证据、RAG 追问、学习任务、训练复盘和后台人工评分串成闭环。当前系统已经具备授权样本沉淀和评测样本导出能力，这意味着后续每一次真实使用都可以转化为经过授权和人工校验的训练数据。这样的数据闭环，是后续做监督微调、偏好优化和模型迭代的基础。

### 6.4 被问“你们到底有没有微调模型”时的回答

建议回答：

> 当前版本没有虚假宣称已经完成底层模型微调。我们做的是更基础、也更关键的微调准备工作：把真实求职场景中的简历、岗位画像、面试问答、报告结果和人工评分变成可授权、可去标识化、可导出的训练样本。后续只要样本量达到阈值，就可以进入 SFT 或 LoRA/QLoRA 微调实验。相比直接说训练了模型，我们更重视数据质量、授权合规和评测闭环。

如果想更有气势：

> 微调不是简单跑一条训练命令，而是先要有领域数据、标注规范和评测基准。我们现在已经把这三个前置条件产品化。

## 7. 评委关注点对齐

| 评委关注 | 推荐回答 |
|---|---|
| 创新性 | 不是通用聊天机器人，而是就业能力诊断、岗位画像、RAG 追问、人工评分和微调样本闭环 |
| 技术难度 | 涉及 PDF 解析、结构化简历、岗位知识切片、证据评分、RAG、权限授权、样本导出和后台标注 |
| 完成度 | 已有用户端、后台端、PostgreSQL、Redis/Celery、Docker、授权和评测样本入口 |
| 数据价值 | 授权样本可沉淀为训练/评测数据，形成长期数据壁垒 |
| 可持续性 | 每次真实面试都能成为模型优化和岗位画像优化的反馈 |
| 合规性 | 用户不同意数据贡献也能使用核心功能；只有授权样本进入人工评分和评测导出 |

## 8. 需要避免的风险

1. 不要在 PPT 或答辩中写“已完成大模型微调”，除非真的完成训练、保存模型版本、跑对照实验。
2. 不要写“自研大模型”，当前更准确是“基于大模型 API 的应用层智能体与数据闭环”。
3. 不要写“完全匿名化”，当前是“去标识化/脱敏”，且可能保留学校、专业、项目经历等提高数据质量的字段。
4. 不要用未授权真实简历做展示或训练样本。
5. 不要把岗位知识库内容说成候选人真实经历。
6. 不要把 AI 自动分数冒充人工评分。

## 9. 本方案对应的下一步升级建议

当前用户要求“继续验收后移”，因此建议临时定义：

- 阶段 141：大模型微调与答辩展示方案文档收口。
- 阶段 142：微调样本 JSONL 导出与统计看板。
- 阶段 143：微调准备报告自动生成。
- 阶段 144：真实样本达到阈值后再做轻量 SFT/LoRA 实验。

但必须保留底线：

- C1/C2/C3 不删除、不作废。
- 阶段 138/140 的真实闭环仍是最终验收口径。
- 微调路线可以作为答辩亮点，但不能替代真实功能验收。

## 10. 答辩最终包装句

> 职启智评的核心不是“让大模型回答面试问题”，而是把求职能力诊断过程中的每一次简历解析、岗位追问、报告生成、学习任务和人工评分，转化为可授权、可追溯、可评测、可用于后续微调的高质量数据资产。这个闭环让系统具备从工具到平台、从平台到模型优化基础设施的升级潜力。
