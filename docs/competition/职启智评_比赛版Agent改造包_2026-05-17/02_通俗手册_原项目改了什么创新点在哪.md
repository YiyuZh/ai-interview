# 通俗手册：最后跟原本项目做了什么更改？创新点在哪里？

这份手册给项目负责人自己看，也可以作为答辩前的“人话版说明”。

---

## 一句话说明

原项目像是一个“能帮学生模拟面试的 AI 工具”。

改造后要讲成：

```text
一个由多个专业 Agent 协作的就业能力诊断系统。
它先看简历有什么证据，再看目标岗位需要什么能力，然后找出差距，围绕差距追问，最后生成报告、学习任务，并把授权样本沉淀成后续微调数据。
```

---

## 1. 原本项目是什么样

原项目主线大致是：

```text
用户上传简历
-> 系统解析简历
-> 用户选择目标岗位
-> 大模型生成能力诊断
-> 大模型生成模拟面试问题
-> 用户回答
-> 大模型生成报告和学习建议
```

这个已经能用，但答辩时容易被说成：

```text
你只是把简历和岗位发给大模型，让大模型生成了一些内容。
```

---

## 2. 改造后变成什么样

改造后的主线是：

```text
简历证据 Agent：判断简历里哪些能力有证据、哪些只是自称、哪些完全缺失。
岗位画像 Agent：从岗位知识库里提取真实岗位要求。
能力差距 Agent：把候选人证据和岗位要求做对比，生成能力缺口矩阵。
简历润色 Agent：基于证据状态和岗位画像生成“可改但不造假”的岗位化表达建议。
AI 面试官 Agent：不是随便问模板题，而是围绕缺口和证据状态追问。
报告 Agent：把问答和诊断整理成报告。
学习任务 Agent：把短板变成可执行训练计划。
数据治理 Agent：判断这次案例能不能脱敏、授权、沉淀。
Eval Agent：给追问和报告打分，比较普通模型和优化后版本。
SFT Agent：把高质量样本转换成后续微调数据。
```

通俗说：

```text
原来是一个大模型在做所有事；
现在是一个“AI 面试工作组”在协同工作。
```

---

## 3. 最明显的变化

### 变化 1：从“问面试题”变成“证据追问”

普通 AI 可能问：

```text
请介绍一下你的后端项目。
```

职启智评改造后会问：

```text
你简历中提到参与过 Flask 后端接口开发，但没有看到 Redis 缓存或接口性能优化的证据。
请结合一个具体项目说明：你是否处理过接口响应慢或缓存一致性问题？当时的业务场景是什么？你采取了什么方案？最终指标有什么变化？
```

区别：

```text
普通 AI 是泛泛提问。
职启智评是根据岗位要求和简历证据缺口追问。
```

---

### 变化 2：从“单次生成”变成“诊断闭环”

原来：

```text
生成一次报告，结束。
```

现在：

```text
诊断 -> 简历润色 -> 面试 -> 报告 -> 学习任务 -> 授权沉淀 -> 人工评分 -> Eval -> 微调数据 -> 下一轮优化
```

这就是项目的闭环。

---

### 变化 3：从“API 应用”变成“AI 工程系统”

原来评委可能觉得：

```text
你只是调用了 OpenAI-compatible API。
```

现在可以解释：

```text
底层模型负责语言能力，但我们的核心在就业诊断工作流。
我们设计了岗位画像、证据状态、能力缺口、Agent 协作、人工评分、Eval 和后训练数据闭环。
```

---

### 变化 4：从“泛泛改简历”变成“证据约束润色”

普通 AI 可能会把简历写得更漂亮，但也容易写出没有证据的经历。

职启智评的简历润色 Agent 先看证据状态：

```text
已证明的能力：可以强化表达。
间接证明的能力：可以改成更清楚的项目描述。
仅声明的能力：只能提示补充真实证据，不能写成已经做过。
缺失的能力：只能给准备建议或占位表达，不能编造项目。
```

通俗说：

```text
它不是帮用户“编简历”，而是帮用户把真实经历表达得更贴岗位。
```

---

### 变化 5：从“只会演示”变成“能持续变强”

原项目如果没有数据闭环，就很难回答：

```text
系统以后怎么变得更好？
```

改造后回答：

```text
每一次授权后的面试案例，都会经过去标识化、人工评分和质量筛选，进入 Eval 或 SFT/DPO 样本池。
系统后续可以通过 SFT 学输出格式和追问风格，通过 DPO 学好坏偏好，通过 QLoRA 在开源模型上降本训练。
```

---

## 4. 创新点到底在哪里

### 创新点 1：Career-AgentOS 多智能体就业诊断架构

不是一个大模型直接回答所有问题，而是拆成多个 Agent：

```text
简历证据 Agent
岗位画像 Agent
能力差距 Agent
简历润色 Agent
面试官 Agent
报告 Agent
学习任务 Agent
数据治理 Agent
Eval Agent
SFT Agent
```

每个 Agent 有明确职责，最终由 OPC 总控 Agent 协调。

---

### 创新点 2：简历证据链

系统不是只看“简历里写了什么”，而是判断：

```text
这个能力有没有项目证据？
是已经证明，还是间接证明？
只是写了熟悉，还是完全缺失？
```

证据状态建议分为：

```text
已证明
间接证明
仅声明
缺失
冲突/待澄清
```

这让面试追问更有针对性。

---

### 创新点 3：岗位画像驱动的动态面试

系统不是用固定题库，而是结合目标岗位画像。

例如目标岗位是 Python 后端：

```text
接口设计
数据库
Redis
并发
部署
排障
日志
性能优化
```

目标岗位是产品助理：

```text
需求分析
PRD
用户调研
数据指标
竞品分析
沟通协作
```

不同岗位对应不同证据和追问。

---

### 创新点 4：证据约束简历润色

系统不是简单让大模型“优化措辞”，而是把润色纳入同一套证据链：

```text
岗位画像告诉用户应该突出什么。
简历证据告诉系统哪些能写、哪些不能写。
能力差距告诉用户哪些需要补证据。
润色 Agent 输出可修改建议、风险提示和待补证据清单。
```

这让简历润色既有岗位针对性，又能避免把岗位知识库内容误写成候选人真实经历。

---

### 创新点 5：证据追问型 AI 面试官

面试官不是泛泛问：

```text
你有什么项目经验？
```

而是围绕“缺口 + 证据状态 + 岗位要求”追问：

```text
目标岗位要求 Redis 缓存能力，但简历没有相关证据。
请说明你是否在项目中处理过缓存穿透、缓存击穿或数据一致性问题。
```

这比普通 AI 面试题库更像真实面试。

---

### 创新点 6：人类在环的数据闭环

系统不是直接把所有用户数据拿去训练，而是设计了：

```text
用户授权
去标识化
后台人工评分
样本质量筛选
训练集 / 验证集拆分
Eval 对比
```

这个闭环让项目更像真实 AI 产品。

---

### 创新点 7：SFT/DPO/QLoRA 三阶段后训练路线

赛前可以讲：

```text
第一阶段：SFT，让模型学会职启智评的追问格式和证据约束。
第二阶段：DPO，让模型学习哪些追问更像高质量面试官。
第三阶段：QLoRA，迁移到开源模型，降低长期调用和训练成本。
```

通俗解释：

```text
SFT 是教模型照着标准答案学。
DPO 是告诉模型哪个答案更好。
QLoRA 是以后自己部署开源模型时，用更低成本做训练。
```

---

## 5. 答辩时怎么讲“我们做了什么”

推荐讲法：

```text
我们对原项目做了三层升级。

第一层是产品升级：从单纯 AI 面试，升级为简历诊断、能力差距、模拟面试、学习任务和训练复盘的完整闭环。

第二层是架构升级：引入 Career-AgentOS 多智能体架构，把简历证据、岗位画像、能力差距、面试追问、报告生成、数据治理和 Eval 拆成多个专业 Agent 协作。

第三层是模型优化升级：设计了从授权样本、人工评分、SFT Preview 到后续 SFT/DPO/QLoRA 的模型后训练路线，使系统不是一次性生成，而是能持续积累样本并优化。
```

---

## 6. 答辩时怎么讲“创新点”

推荐讲法：

```text
我们的创新不在于重新造一个底层大模型，而在于把大模型能力工程化地嵌入就业诊断场景。

第一，提出 Career-AgentOS 多智能体就业诊断架构。
第二，设计简历证据链，让系统区分已证明、仅声明和缺失能力。
第三，基于岗位画像 RAG 进行动态追问，而不是使用固定题库。
第四，引入人类在环评分和 Eval，让每次面试结果都能沉淀为可评估样本。
第五，规划 SFT、DPO、QLoRA 的后训练路线，让系统可以持续优化。
```

用户能看到：

```text
更精准的能力诊断
更有针对性的面试追问
更清晰的报告和学习任务
```

```text
样本授权状态
样本质量评分
Agent Trace
Eval 评分
SFT Preview 数据
```

PPT 能看到：

```text
多 Agent 架构图
业务闭环图
证据链示例
岗位画像示例
追问对比示例
Eval 对比表
三个月后训练路线图
```
