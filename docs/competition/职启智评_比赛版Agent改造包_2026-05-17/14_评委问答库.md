# 评委问答库

## Q1：你是不是只是套了一个大模型 API？

推荐回答：

```text
底层语言能力确实来自大模型 API，但我们的核心不是简单聊天调用，而是把大模型嵌入到就业能力诊断流程里。

系统前面有岗位画像和简历证据链，中间有能力差距诊断、证据约束简历润色和证据追问型面试，后面有报告生成、学习任务、人类在环评分、Eval 和后训练样本沉淀。

所以我们做的是一个场景化的 AI 工程系统，而不是一个简单问答壳。
```

---

## Q2：你的创新点是什么？

推荐回答：

```text
我们有五个创新点。

第一，Career-AgentOS 多智能体就业诊断架构。
第二，简历证据链，区分已证明、间接证明、仅声明和缺失。
第三，岗位画像 RAG 驱动动态追问，而不是固定题库。
第四，证据约束简历润色，帮助学生优化真实经历表达但不编造经历。
第五，人类在环评分和 Eval，让输出质量可评估。
第六，SFT、DPO、QLoRA 三阶段后训练路线，让系统能够持续优化。
```

## Q3：微调到底完成了吗？

比赛版推荐回答：

```text
我们当前已经完成面向面试追问任务的后训练方案设计和工程化入口，包括 SFT-ready 数据结构、样本准入流程、Eval 评分维度和训练 Preview。

赛前版本重点展示多 Agent 编排和 SFT Preview；三个月内会接入真实授权样本，完成 OpenAI SFT job、holdout eval，并进一步扩展到 DPO 偏好对齐和 QLoRA 开源模型降本验证。
```

如果已经真实跑完 job，再升级为：

```text
【仅 job succeeded + fine_tuned_model 后可用】我们已经完成一次 OpenAI SFT 实验，并保存了 job id、fine_tuned_model、样本统计和 holdout eval 记录。
```

## Q4：为什么第一版只训练面试追问，不训练完整报告？

推荐回答：

```text
因为面试追问是更窄、更可控、更容易评估的任务。

它的输入是简历摘要、目标岗位、能力缺口、证据状态和面试上下文；输出是下一轮追问，结构清晰，适合人工评分和 SFT。

完整报告生成输出太长，变量太多，小样本阶段容易互相干扰，所以我们先把追问任务做稳，再扩展到报告生成。
```

---

## Q4-补充：简历润色会不会帮学生造假？

推荐回答：

```text
不会。我们的简历润色 Agent 是证据约束的，不是自由发挥式润色。

系统会先判断能力证据状态：已证明的能力可以强化表达，间接证明的能力可以补清楚场景和动作，仅声明或缺失的能力只能提示补充真实证据，不能写成已经做过。

例如简历没有 Redis 项目时，系统不会写“负责 Redis 缓存优化”，只会提示用户准备真实缓存或性能优化案例。
```

## Q5：为什么不用 LoRA 或 QLoRA 作为第一版？

推荐回答：

```text
LoRA 和 QLoRA 更适合在本地或自托管开源模型上做参数高效微调。

我们当前第一阶段是 API-only 工程路线，所以优先做 OpenAI SFT 和数据闭环，保证开发速度和答辩效果。

后续三个月我们会把 QLoRA 放到开源模型私有化验证阶段，用来降低长期调用成本和提升可控性。
```

---

## Q6：DPO 在这里有什么用？

推荐回答：

```text
SFT 主要让模型学会标准追问格式和业务口径，而 DPO 可以让模型进一步学习“什么追问更好”。

例如同一个能力缺口下，一个问题只是泛泛问“你了解 Redis 吗”，另一个问题要求候选人结合项目说明缓存一致性、方案和指标。我们会把后者作为 chosen，前者作为 rejected，形成偏好对齐数据。
```

---

## Q7：你怎么证明系统变好了？

推荐回答：

```text
我们设计了 Eval Judge 评分体系，从能力缺口聚焦、证据约束、追问深度、润色可执行性、岗位贴合、格式稳定和可用于报告七个维度评分。

赛前用三岗位沙盘样例展示 baseline_prompt_preview 和 Agent optimized 的规则评分差异；这里的 baseline 是 Preview 规则基线，不是真实模型实测。后续接入真实授权样本后，会用 holdout 对比 base model、fine-tuned model 和 Agent 版本。
```

---

## Q8：数据从哪里来？

推荐回答：

```text
数据来源分为两类。

第一类是真实授权样本，来自用户完整训练闭环，经过去标识化和人工复核后进入评估或训练样本池。

第二类是构造样本，用来补齐格式、岗位覆盖和边界情况，主要用于演示和早期 schema 验证。

未来真实训练效果以授权样本和 holdout eval 为准。
```

---

## Q9：一个人怎么维护这么多 Agent？

推荐回答：

```text
这里的 Agent 不是指一开始就部署大量独立服务，而是一套可演进的职责拆分。

赛前阶段是文档型和脚本型 Agent，用于明确流程、生成 trace 和演示资产。

后续阶段再逐步服务化，把核心 Agent 接入 FastAPI 后端。因此一人项目也可以通过 OPC 总控模式逐步推进。
```

## Q10：这个项目的商业或落地价值是什么？

推荐回答：

```text
它可以服务高校就业指导、学生求职训练、职业规划课程和企业校招准备。

相比普通 AI 面试工具，职启智评能把简历证据、岗位要求、面试反馈和学习任务串成闭环，帮助学生明确自己与目标岗位之间的差距，并持续训练补强。
```
