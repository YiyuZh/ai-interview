## 2. 适用范围与边界

本方案适用于职启智评平台中与求职训练有关的个人信息处理活动，包括账号登录、简历上传、简历解析、岗位画像匹配、能力诊断、简历润色、学习任务、模拟面试、报告生成、训练复盘、后台人工评分和评测数据沉淀。

当前项目的主链路为：

```text
简历解析 -> 目标岗位 -> 岗位画像/岗位知识库 -> 简历证据评分 -> 能力差距诊断 -> 学习任务 -> AI 模拟面试 -> 报告 -> 训练复盘 -> 人工评分沉淀
```

本方案不表示以下事项已经完成：

- 不表示项目已经完成正式个人信息保护影响评估。
- 不表示 C1/C2/C3 三岗位真实闭环已经全部通过。
- 不表示前端已经上线独立隐私政策勾选框、账号注销入口或数据下载入口。
- 不表示项目已经训练自研大模型或完成官方微调。
- 不表示第三方大模型供应商的数据留存、训练使用和跨境安排已经全部完成法务审查。

## 3. 用户侧隐私协议初版

### 3.1 我们为什么处理个人信息

职启智评用于帮助学生和求职者进行就业能力诊断与训练。平台处理个人信息的目的包括：

1. 完成账号注册、登录、身份识别和权限控制。
2. 解析用户上传的简历 PDF，提取教育经历、实践经历、项目经历、技能证书和求职意向等信息。
3. 根据目标岗位画像生成能力诊断、证据评分、能力差距和学习任务。
4. 基于简历证据、岗位要求和训练目标生成模拟面试问题。
5. 根据用户回答生成面试报告、训练复盘和后续改进建议。
6. 支持后台人工评分、AI 与人工评分对比、系统质量验证和比赛答辩材料准备。
7. 保障系统安全、排查故障、防止越权访问和异常调用。

### 3.2 我们可能收集和处理的信息

| 信息类别 | 示例字段 | 使用场景 | 敏感程度 | 当前处理建议 |
|---|---|---|---|---|
| 账号与登录信息 | 用户名、邮箱、手机号、密码哈希、登录 token、角色权限 | 注册、登录、权限判断 | 一般个人信息；密码相关信息需重点保护 | 密码不得明文存储；token 设置有效期 |
| 后台管理员信息 | 管理员邮箱、角色、`can_manage_admins`、`is_root_admin`、操作记录 | 后台管理、人工评分、权限审计 | 一般个人信息与权限数据 | root 管理员不可删除、禁用或收回核心权限 |
| 原始简历文件 | PDF 简历、文件名、上传时间 | 简历解析、训练材料来源 | 可能包含敏感个人信息 | 仅在授权后上传；不得随意放入公开仓库 |
| 简历解析文本 | 教育、经历、项目、技能、证书、联系方式等抽取文本 | 能力诊断、面试出题、报告生成 | 可能包含敏感个人信息 | 最小必要使用；展示和导出时优先脱敏 |
| 结构化简历 | `normalized_resume`、`parse_quality`、`extraction_diagnostics` | 提高解析稳定性和开始面试可靠性 | 个人信息派生数据 | 与原始简历绑定权限控制 |
| 目标岗位与画像 | 目标岗位、岗位能力、知识切片、追问方向 | 岗位匹配、RAG 出题、简历润色 | 通常不属于个人信息 | 不得写成候选人真实经历 |
| 能力评价快照 | `resume_evaluation_snapshot`、证据状态、能力缺口、评分 | 统一诊断、润色、学习计划、面试和报告口径 | 个人评价类数据 | 应可解释、可复核，不得过度推断 |
| 学习任务与复盘 | 学习任务、完成状态、笔记、下一轮目标 | 求职训练闭环 | 个人成长记录 | 仅用于当前用户训练和授权评估 |
| 面试过程数据 | 面试问题、用户回答、AI 评价、轮次记录 | 模拟面试、报告、人工评分 | 个人表现数据 | 不用于真实招聘筛选决策 |
| 报告与评分数据 | 综合分、能力短板、报告建议、人工评分、评分备注 | 训练反馈、系统验证、比赛材料 | 个人评价类数据 | 人工评分需有依据，禁止伪造 |
| 评测与导出数据 | JSONL、ZIP、评测样本、AI/人工差异 | 系统评测、后续校准或微调准备 | 可能包含个人信息 | 导出前脱敏；未授权不得外发 |
| 系统日志 | 请求路径、错误日志、IP、设备信息、时间戳 | 安全审计、故障排查 | 一般个人信息或设备标识 | 设定留存期限，限制访问 |
| 第三方模型调用数据 | 提示词、简历片段、岗位信息、回答内容、模型返回结果 | AI 解析、面试、报告生成 | 可能包含个人信息 | 最小化发送；披露供应商和风险 |

### 3.3 简历属于重点保护数据

用户简历可能包含姓名、电话、邮箱、学校、专业、教育经历、实习经历、项目经历、证书、获奖情况、社团经历和求职意向等信息。部分简历还可能包含照片、政治面貌、身份证明、住址、健康情况或家庭信息。平台应按重点保护数据处理简历，不应把简历当作普通文本随意复制、传播、公开展示或提交到不明第三方服务。

阶段 138 真实案例跑测时，每个案例必须记录简历来源和授权情况。允许使用的来源包括：

- 用户本人授权提供的真实简历。
- 来源清楚且允许使用的公开授权简历样例。
- 团队自行构造的合成案例。

不允许使用的来源包括：

- 未经本人同意从社交平台、网盘、聊天记录、招聘网站等渠道取得的真实简历。
- 来源不明、授权不明或无法说明用途边界的简历。
- 含有明显无关敏感信息且未做脱敏处理的简历。

### 3.4 授权同意与最小必要原则

用户上传简历、选择目标岗位、开始模拟面试或提交训练复盘前，应理解平台会基于其输入内容生成诊断、问题、评分和报告。后续产品化时，建议在以下节点增加清晰授权提示：

1. 注册或首次登录时展示隐私协议与用户协议。
2. 上传简历前提示简历数据用途、第三方模型调用风险和删除方式。
3. 将案例加入后台人工评分或评测数据集前，取得额外授权或完成充分脱敏。
4. 使用境外大模型 API 或可能发生个人信息出境时，另行告知并履行适用规则。

系统应遵循最小必要原则。能够通过岗位画像、能力项和脱敏片段完成任务时，不应把完整简历全文发送给无关组件或第三方。

基础隐私协议同意用于提供简历解析、能力诊断、简历润色、模拟面试、报告、训练复盘、账号安全和故障排查等核心功能。该部分属于平台提供服务所必需的个人信息处理，不同意时平台无法完成核心训练链路。

去标识化数据贡献授权应作为单独选项，默认不勾选；用户不同意数据贡献授权时，仍可继续使用简历解析、润色、模拟面试、报告和训练复盘等核心功能。产品文案可以使用“推荐开启数据贡献计划”“授权后我们可以基于真实使用数据发现简历解析、岗位画像、面试追问、评分规则、报告建议和学习任务中的问题，针对性升级项目，让后续诊断和训练效果更贴近真实求职场景”等正向说明，但不得隐藏拒绝选项、默认勾选、反复骚扰或将非必要授权与核心服务捆绑。

### 3.5 去标识化数据贡献授权

用户可以自愿选择是否允许平台将去标识化后的简历解析结果、学校/专业/项目或实习经历、目标岗位、面试问答、报告和人工评分用于系统评测、比赛材料、质量改进和数据集沉淀。该授权不应默认勾选，不应与核心服务捆绑强制同意。

如果用户允许平台使用上述去标识化数据，项目团队可以基于真实闭环样本定位高频问题，例如简历解析遗漏、岗位画像不够贴合、能力差距识别偏差、面试追问不够深入、评分规则与人工评分不一致、报告建议不够具体、学习任务不可执行等。团队可据此定向升级解析规则、岗位知识切片、证据评分、RAG 追问策略、报告模板和学习任务生成逻辑，从而让后续用户获得更准确、更稳定、更贴近真实求职场景的诊断与训练效果。

去标识化处理应优先删除或遮挡姓名、手机号、邮箱、证件号、学号、详细住址、文件名中的个人标识等直接识别信息。为保证岗位画像匹配、能力诊断、人工评分和评测数据质量，平台可以在明确告知并获得用户同意后保留学校层级或学校名称、专业、教育经历、实习经历、项目经历、技能、目标岗位、面试问答、报告和评分信息。该处理不等同于完全匿名化，不得对外宣称“无法识别任何个人”。

除账号级默认授权外，平台可以在报告页或训练复盘页提供“本次案例加入去标识化评测库”的单独授权入口。该入口适合在用户看到报告价值后出现，有利于获得更高质量的真实闭环样本；但本次案例授权只影响对应面试记录，不应自动扩大到其他简历、其他面试或账号全局授权。

用户撤回数据贡献授权后，后续新上传简历和新面试默认不再进入评测、比赛展示和数据集沉淀流程。对于撤回前已经按照授权形成的去标识化材料，应按项目归档、安全管理和后续合规要求处理，不应在页面或文档中承诺立即删除所有历史材料。

### 3.6 第三方大模型与外部服务

职启智评当前技术路线包含 DeepSeek、OpenAI-compatible API 等大语言模型接口。第三方模型可能参与简历解析、面试问题生成、报告生成或文本润色。

后续正式上线前，应补齐第三方服务清单，至少包括：

| 服务类型 | 可能用途 | 需披露内容 |
|---|---|---|
| 大语言模型 API | 简历解析、问题生成、报告生成、润色建议 | 供应商名称、处理目的、发送字段、是否用于训练、保存期限、接口区域 |
| OCR 组件 | 低质量 PDF 或图片型简历文本抽取 | 是否本地处理、是否上传第三方、是否默认开启 |
| 云服务器与数据库 | 部署应用、保存业务数据 | 服务器区域、数据库类型、备份策略、访问控制 |
| 邮件或短信服务 | 注册、通知、找回密码 | 发送字段、供应商、保存期限 |

当前 OCR 不应默认全量开启，应通过 `ENABLE_RESUME_OCR=true` 控制。若未来引入语音、视频、摄像头、麦克风或屏幕录制，应单独更新隐私协议并取得明确同意。

### 3.7 用户权利

用户依法享有对个人信息的查阅、复制、更正、补充、删除、撤回同意、注销账号、投诉举报等权利。后续产品化时，建议提供以下入口：

- 个人中心查看简历、诊断、面试记录、报告和训练复盘。
- 删除已上传简历、面试记录或训练记录。
- 申请导出个人训练记录。
- 撤回将案例用于人工评分、评测数据集或比赛展示的授权。
- 注销账号并删除或匿名化关联数据。
- 通过隐私邮箱或校内负责人提交投诉、咨询和更正请求。

在当前比赛和测试阶段，如果系统页面尚未提供自助入口，应通过项目负责人或管理员人工处理，并记录处理时间、处理人、处理结果和无法立即删除的原因。

### 3.8 保存期限

在没有正式运营制度前，建议采用保守留存口径：

| 数据类别 | 建议保存期限 | 到期处理 |
|---|---:|---|
| 原始简历 PDF | 测试完成后 180 天内，或用户要求删除时 | 删除原文件；确需保留时做脱敏说明 |
| 结构化简历与评价快照 | 测试完成后 1 年内，或账号删除时 | 删除或匿名化 |
| 面试问答、报告、学习任务、训练复盘 | 测试完成后 1 年内，或用户要求删除时 | 删除或匿名化 |
| 后台人工评分与评测样本 | 项目评测期内保留；进入长期数据集前必须脱敏 | 去标识化、编号化、删除联系方式 |
| 系统日志 | 180 天内，安全事件调查需要时可延长 | 到期清理或归档脱敏 |
| 备份数据 | 按备份周期覆盖，建议不超过 180 天 | 到期自动覆盖或删除 |

如法律法规、学校管理要求或比赛材料归档要求另有规定，应按更严格要求执行，并在隐私协议中说明。

### 3.9 自动化评分与人工复核

平台生成的能力评分、短板、面试评价和学习建议用于训练，不作为真实招聘录用、淘汰或处分依据。系统应避免仅基于自动化评分对用户作出具有重大影响的决定。

后台人工评分用于验证系统质量和沉淀评测数据。人工评分应记录评分依据、分项分数和备注，不能伪造评分，不能把 AI 自动评分冒充人工评分。

### 3.10 争议、投诉与诉讼风险处理

如用户认为其个人信息权益受到影响，可以通过项目公布的联系方式提出查询、更正、删除、撤回授权或投诉请求。项目团队应在合理期限内响应。

如发生个人信息泄露、滥用、未经授权公开、超范围处理、第三方违规使用等情况，应立即停止相关处理，隔离风险数据，保留日志，评估影响范围，并根据适用法律法规和学校管理要求履行告知、报告和整改义务。

本协议不限制用户依法向有关主管部门投诉、举报或依法提起诉讼的权利。

## 4. 内部个人信息处理方案

### 4.1 数据生命周期

| 阶段 | 输入 | 处理动作 | 输出 | 控制要求 |
|---|---|---|---|---|
| 采集 | 账号信息、简历 PDF、目标岗位 | 用户主动提交 | 上传记录、目标岗位 | 明示用途，记录来源和授权 |
| 解析 | PDF 文本、OCR 结果 | 文本抽取、结构化、质量诊断 | `normalized_resume`、`parse_quality` | OCR 默认关闭，失败可解释 |
| 评价 | 结构化简历、岗位画像 | 证据评分、能力差距分析 | `resume_evaluation_snapshot` | 证据状态统一，避免过度推断 |
| 润色 | 简历证据、岗位要求 | 给出表达建议 | 润色建议 | 不编造经历、公司、时间、指标 |
| 学习 | 能力缺口 | 生成学习任务 | 任务、产出物、验收方式 | 任务可编辑、可保存 |
| 面试 | 简历证据、能力缺口、岗位切片 | 生成问题、记录回答 | 问答记录、验证目标 | 追问来源可解释 |
| 报告 | 问答记录、评价快照 | 生成总结与建议 | 面试报告 | 区分已证实、待验证、缺失 |
| 复盘 | 用户笔记、下一步目标 | 保存训练复盘 | 复盘记录 | 用户可查看和删除 |
| 人工评分 | 问答、报告、任务、证据 | 后台标注 | 人工分、备注、差异原因 | 至少记录评分依据 |
| 数据沉淀 | 脱敏样本、评分记录 | 导出评测数据 | JSONL、CSV、ZIP | 授权、脱敏、审计 |
| 删除归档 | 到期数据或用户请求 | 删除、匿名化、备份覆盖 | 删除记录 | 可追溯处理结果 |

### 4.2 权限与角色控制

| 角色 | 可访问内容 | 不应访问内容 | 控制要求 |
|---|---|---|---|
| 普通用户 | 本人简历、诊断、面试、报告、学习任务、复盘 | 他人简历和后台评分 | 仅本人数据范围 |
| 普通管理员 | 后台被授权查看的案例和评分页面 | 管理员授权、删除 root、越权导出 | 最小权限 |
| 授权管理员 | 管理员列表、授权范围内的用户与案例 | root 权限收回、无关原始简历批量下载 | 操作记录和审批 |
| root 管理员 | 初始化管理、权限配置、关键排障 | 不必要的个人内容浏览 | root 不可删除、禁用或收回权限 |
| 数据标注人员 | 脱敏后的案例、问答、报告、评分表 | 原始联系方式、未授权简历全文 | 标注前脱敏 |
| 开发人员 | 测试样例、日志、错误堆栈 | 生产真实简历和明文 token | 使用最小化测试数据 |

仓库中不得保存真实密码、生产 API Key、数据库密码、未脱敏真实简历、未授权导出的 JSONL 或包含联系方式的测试包。

### 4.3 阶段 138 真实闭环专项要求

阶段 138 的目标是完成 C1/C2/C3 三岗位真实闭环验收和人工评分沉淀。隐私与数据处理要求如下：

1. 每个案例开始前，记录简历来源、授权情况、目标岗位、模型供应商、模型名称、服务器 commit 和测试人员。
2. C1 Python 后端开发工程师可以使用公开授权样例或合成样例，必须记录来源链接或构造说明。
3. C2 产品助理/产品经理实习生如使用 `2590603008詹已誉简历.pdf`，必须确认其属于用户本人授权材料，仅用于本项目测试和比赛材料。
4. C3 人力资源专员如暂无真实授权简历，应使用合成案例或公开授权样例，不得临时抓取真实求职者简历。
5. 每个案例必须完成上传简历、能力诊断、简历润色、至少一个学习任务、至少 3 轮面试、报告、训练复盘和后台人工评分。
6. CSV 与 Markdown 记录不得伪造通过状态。未跑测、待收集、失败或仅完成静态诊断的案例，应如实标记。
7. 人工评分至少记录分项分数、评分理由和 AI/人工差异原因。若多人评分，记录平均分和差异说明。
8. 用于答辩展示的截图应遮挡姓名、电话、邮箱、学校编号、住址、证件号等可识别信息。

### 4.4 岗位知识库与候选人经历的边界

岗位知识库、成员补充资料、面经和 RAG 切片只能作为以下用途：

- 岗位要求参考。
- 面试追问方向。
- 简历润色表达建议。
- 能力项和评分规则参考。
- 学习任务设计参考。

不得将岗位知识库内容写成候选人的真实经历。若简历只写“熟悉 Redis / SQL”等声明，应标记为待验证或 `claimed_only`，通过面试追问核实，不得直接判定为高水平掌握。

### 4.5 数据脱敏规则

进入比赛材料、评测数据集、导出包或公开演示前，建议至少脱敏以下信息：

| 类型 | 脱敏方式 |
|---|---|
| 姓名 | 替换为候选人 A、R1 用户、测试用户 |
| 手机号 | 保留后四位或全部替换为 `***` |
| 邮箱 | 用户名前缀和域名部分遮挡 |
| 学号、身份证、证书编号 | 全部遮挡 |
| 住址、家庭信息 | 删除 |
| 学校、公司 | 数据贡献授权场景下可为评测质量保留学校层级/学校名称、专业和经历背景；公开展示时再按需要泛化为某高校、某公司 |
| 文件名 | 去除真实姓名、手机号等标识 |
| 截图 | 遮挡头像、联系方式、账号、token、URL 参数 |
| 日志 | 删除 token、API Key、Cookie、Authorization header |

脱敏后仍能通过多项信息组合识别个人时，应继续泛化或改用合成样例。

### 4.6 数据集与导出控制

后台评测样本、JSONL、ZIP、CSV 或 Excel 导出文件可能包含简历、回答、评分和备注，应作为受控数据处理：

1. 只有授权管理员或数据负责人可以导出。
2. 导出前应选择脱敏字段，不默认导出原始联系方式。
3. 导出文件命名应包含日期、用途、脱敏状态和负责人。
4. 导出记录应保存操作人、时间、用途、文件范围和是否脱敏。
5. 未经授权不得上传到公开 Git 仓库、网盘公开链接、公开论坛或第三方训练平台。
6. 当前阶段只能表述为“微调准备数据”或“fine-tune-ready data”，不得声称已经完成大模型微调。

### 4.7 第三方大模型调用控制

调用第三方大模型前，应尽量减少发送内容：

- 能发送能力项和脱敏片段时，不发送完整简历。
- 能发送岗位能力摘要时，不发送完整岗位知识库。
- 能在本地完成规则评分时，不把评分表和全部原文发送给模型。
- 不在 prompt 中包含 API Key、数据库连接串、后台 token 或管理员密码。
- 对模型返回结果进行结构化校验，避免把模型编造内容写入候选人经历。

后续上线前，应逐一确认第三方模型供应商是否会保存请求、是否用于训练、数据存储区域、删除机制、子处理方和安全措施。若涉及向境外提供个人信息，应按适用法律法规另行评估并履行个人信息出境相关要求。

### 4.8 安全控制措施

当前项目应至少保持以下安全控制口径：

1. 生产环境使用 HTTPS 或受控反向代理，避免明文传输简历和 token。
2. 密码只保存哈希，不保存明文密码。
3. API Key、数据库密码、root 管理员初始密码通过环境变量配置，不写入仓库。
4. PostgreSQL、Redis、Celery 和对象文件目录限制公网暴露。
5. 后台管理员使用最小权限和明确角色。
6. 关键导出、删除、授权、人工评分操作应留痕。
7. 服务器日志避免输出完整简历、完整 prompt、token 和密钥。
8. 定期执行数据库容量、自检脚本和异常日志检查。
9. 对未跟踪大文件、申报材料压缩包、成员原始资料保持谨慎，不默认提交。

### 4.9 安全事件响应

若发生疑似个人信息安全事件，按以下流程处理：

1. 发现：记录发现时间、发现人、影响系统、初步现象。
2. 隔离：暂停相关接口、导出链接、账号或第三方调用。
3. 评估：确认涉及数据类别、人数、泄露范围、是否含敏感个人信息。
4. 补救：删除公开文件、撤回链接、重置密钥、修复权限或代码问题。
5. 告知：根据适用法律法规、学校管理要求和用户影响程度告知相关个人或主管部门。
6. 复盘：形成事件报告，补充权限、脱敏、日志和测试措施。
