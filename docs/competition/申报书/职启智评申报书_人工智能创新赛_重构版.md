## 三、选题需求

高校毕业生求职准备存在一个长期痛点：学生知道自己要找工作，但往往不知道目标岗位具体要求什么、自己的能力差在哪里、应该先补哪项能力，也缺少可反复训练和复盘的工具。就业指导老师可以提供人工咨询，但面对大量学生时，难以做到高频、个性化、可留痕、可量化的持续辅导。

从就业形势看，高校毕业生规模持续处在高位。公开报道显示，2026 届高校毕业生预计规模约 1270 万人，求职竞争和就业指导压力仍然较大。单纯告诉学生“优化简历”“多练面试”并不足够，学生更需要一个能把“目标岗位要求”和“个人现有证据”对齐的系统，告诉他：

1. 当前简历能支持哪些能力结论。
2. 哪些能力只是写了关键词，但缺少项目或结果证据。
3. 哪些岗位关键能力需要通过面试追问验证。
4. 哪些短板应该优先学习和训练。
5. 学完、练完之后如何复盘下一步。

因此，本项目的选题需求不是再做一个普通 AI 聊天式面试工具，而是面向大学生求职准备，构建“简历解析 -> 岗位能力图谱 -> 能力差距诊断 -> 学习任务 -> 模拟面试 -> 训练复盘”的就业能力提升闭环。

## 四、市场现状与相关产品缺陷

目前市场上已经有不少 AI 面试、简历优化和求职辅导产品，例如 AI 模拟面试工具、简历润色工具、面试题库平台和求职训练网站。这些产品说明 AI 已经进入求职辅导场景，但也暴露出几个共性不足：

1. 许多工具偏向一次性问答。用户输入简历或岗位后，系统生成若干面试题或建议，但缺少长期学习任务和训练复盘。
2. 许多产品更像“AI 聊天窗口”。它们能生成文字，但没有把岗位要求拆成能力模型，也没有明确说明能力诊断依据。
3. 题库化倾向明显。部分工具给用户刷题或模拟问答，但问题未必根据用户简历证据、岗位能力缺口和面试表现动态调整。
4. 证据约束不足。简历里写了某个技能，系统可能直接相信；简历里没有完全命中关键词，系统又可能误判为完全缺失，容易出现“过度相信”和“死扣关键词”两类问题。
5. 缺少后台可维护知识资产。岗位要求、企业侧重点、真实面经、学习路线和人工评分样本如果无法持续维护，系统很难从演示工具成长为可运营平台。
6. 缺少 AI 与人工评分对照。就业能力评价涉及真实求职判断，如果没有人工评分、案例标注和重复运行记录，就难以证明 AI 结论稳定、可信、可复核。

这些缺陷说明，单纯接入大模型 API 不是核心竞争力。真正需要补的空白是：把大模型能力放入一个可解释、可维护、可复盘的就业能力诊断系统中。

## 五、本项目填补的空白

职启智评试图填补“AI 面试生成”和“真实就业能力提升”之间的断层。项目不是让 AI 随机生成面试题，而是先把岗位要求结构化，再把简历证据结构化，最后把两者的差距转化为可训练任务。

本项目填补的空白主要体现在四个方面：

1. 从岗位名称走向岗位能力图谱。系统把目标岗位拆成核心能力、要求等级、权重、关键词、证据线索和可提升系数，避免只按岗位名称泛泛分析。
2. 从关键词匹配走向证据分层。系统区分直接证据、间接证据、声明待验证和暂无证据，不把“简历写了掌握”直接等同于“真正掌握”。
3. 从面试问答走向学习闭环。系统不仅生成面试问题，还能把能力缺口转为学习任务、学习路线和训练复盘。
4. 从单次演示走向数据沉淀。后台支持岗位知识库、结构化面试问答经验、学习路线管理、案例标注和人工评分，为后续评估和数据集建设打基础。

## 六、内容与功能

### 6.1 用户端功能

用户端面向学生和求职训练用户，主要包括：

1. 简历上传与解析：上传 PDF 简历，提取教育经历、项目经历、实习经历、技能和证书等结构化信息。
2. 目标岗位选择：选择或输入目标岗位，系统读取公共岗位画像或用户私有岗位画像。
3. 能力诊断：展示岗位匹配度、能力差距、简历证据、待验证能力和优先补强项。
4. 学习任务：把能力缺口转成学习任务，支持任务完成状态、笔记、薄弱项变化和导入导出。
5. 学习计划生成：根据后台基础学习路线模板、成熟学习计划、缺失能力和用户补充资料生成可编辑学习计划。
6. 模拟面试：围绕目标岗位、简历证据、岗位知识切片和待验证能力生成追问。
7. 面试报告：输出优势、短板、回答质量、证据链、改进建议和下一步训练方向。
8. 训练复盘：记录用户自评、复盘笔记、下一轮目标，形成持续训练闭环。
9. 私有岗位画像：用户可维护自己的岗位画像和问答经验，公共画像只读。

### 6.2 后台管理功能

后台面向项目维护者和运营成员，主要包括：

1. 公共岗位画像库：维护岗位要求、问答经验、能力模型、面试追问四类分区。
2. 结构化面试问答经验：维护真实问题、参考回答要点、复盘反思、公司场景、考察能力、难度、来源说明和审核状态。
3. 学习路线管理：维护基础学习路线模板和成熟学习计划，支持筛选、复制、导入导出、质量审核、任务预览和覆盖矩阵。
4. 案例标注工作台：集中记录 R1-R5 等真实案例的简历来源、人工评分、证据对齐、问题质量、报告可执行性和数据集归档状态。
5. 测评样本与人工评分对比：将 AI 输出与人工评价进行对照，作为后续系统校准依据。

### 6.3 技术流程

项目技术链路如下：

```text
Vue 用户端 / Vue 管理端
    -> FastAPI 后端接口
    -> PostgreSQL / Redis / Celery / Docker
    -> 简历解析、岗位画像、能力图谱、知识切片、学习任务、训练复盘
    -> DeepSeek / OpenAI-compatible API 等大模型能力
    -> 证据约束、RAG 多信号路由、规则评分与报告生成
```

大模型在本项目中承担自然语言理解、内容组织和生成辅助能力；项目自身的核心工作在于岗位能力图谱、证据分层、能力差距模型、学习任务闭环、后台知识资产和人工评分体系。

## 七、创新点

### 7.1 岗位能力图谱

系统不直接让 AI 判断“适不适合某岗位”，而是先将岗位拆解为能力项、等级要求、权重、关键词、证据线索和可提升系数。这样可以把岗位要求变成可计算、可解释、可维护的数据结构。

### 7.2 能力差距量化模型

系统根据简历证据和目标岗位要求计算能力差距，并按照“岗位重要性 × 能力差距 × 可提升系数”排序，给出优先补强项。这样避免 AI 只给出泛泛建议。

### 7.3 证据分层与面试验证优先

系统将能力证据分为直接证据、间接证据、声明待验证和暂无证据。对于“简历写了掌握但没有项目支撑”的能力，系统不会直接高分，而是生成验证型追问；对于“同义表达但未完全命中关键词”的能力，系统不会简单判缺失。

### 7.4 RAG 多信号路由与幻觉约束

系统在岗位知识切片检索中引入多信号排序，包括字面匹配、技能主题匹配、薄弱项匹配、来源质量和场景匹配，并将证据置信度传给问题生成与报告生成。岗位知识库只用于校准问题方向，不能被当作用户真实经历，从而减少模型幻觉。

### 7.5 学习任务与训练复盘闭环

项目将能力差距转化为学习计划、练习任务和验收方式。用户完成学习后可以继续面试训练，再通过报告和训练复盘更新下一轮目标，形成持续提升闭环。

### 7.6 后台可运营知识资产

项目不仅有用户端展示，还提供后台维护能力：岗位画像、面试问答经验、学习路线、案例标注、测评样本和人工评分对比都能持续积累。这使项目具备从演示系统走向长期维护平台的基础。

## 八、当前建设基础与边界说明

当前项目已经完成多项本地初版能力，包括：

1. 多岗位画像与岗位能力图谱初版。
2. 简历能力诊断与证据分层。
3. 学习任务中心与学习计划生成流程。
4. 前后台结构化面试问答经验库。
5. 后台学习路线管理、质量审核、覆盖矩阵。
6. 训练复盘和学习任务服务器持久化设计。
7. RAG 多信号路由和证据约束初版。
8. 五人维护资料夹、完整面试闭环案例模板和数据标注口径。

需要明确的是，当前项目不宣称已经自研底层大模型，不宣称真实微调已经上线，不宣称已完成商业化落地。项目创新集中在应用层系统设计：岗位能力建模、证据约束、RAG 路由、学习闭环、后台知识资产和人工评分数据沉淀。

当前仍待补强的部分包括：

1. 服务器统一验收和最新迁移执行。
2. R1-R5 真实完整面试闭环案例。
3. 人工评分与 AI 输出对比数据。
4. 12 个岗位的高质量岗位要求和面经内容补全。
5. 学习任务质量人工抽查。

## 九、预期成果

项目完成后，预期形成以下成果：

1. 一个可运行的就业能力诊断与提升平台。
2. 一套覆盖多岗位的岗位能力图谱和岗位知识库。
3. 一套可解释的能力差距诊断与学习任务生成机制。
4. 一套面向真实面试的证据追问和训练复盘流程。
5. 一批经过人工审核的真实案例、问答记录和评分样本。
6. 一套可支撑后续答辩、论文、数据集和模型校准的过程资料。

## 十、后续计划

后续工作按优先级推进：

1. 完成服务器统一验收，确认用户端、后台端、健康接口、岗位画像、学习路线、案例标注和测评样本均可正常访问。
2. 用 R1-R5 授权案例跑通完整闭环，每个案例至少包含简历、能力诊断、3-5 轮问答、报告、复盘和人工评分。
3. 补全岗位知识库，优先补 Python、Java、前端、测试、产品、运营、HR 等岗位的岗位要求、公司侧重点和结构化问答经验。
4. 抽查学习任务质量，确保任务具备学习材料、练习任务、产出物和验收方式。
5. 根据真实案例更新项目说明书、答辩材料和后续申报材料。
