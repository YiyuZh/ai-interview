# 职启智评数据集与岗位知识库建设手册

适用项目：`D:\apps\ai-interview`

本手册解决三个问题：数据集是什么、去哪里找、怎么放进当前项目。先记住一句话：

```text
岗位知识库负责“面试官该问什么、按什么标准问”；
数据集负责“系统表现好不好、评分准不准、后续能不能校准或微调”。
```

## 1. 数据集到底是什么

在本项目里，数据集分 4 类。

| 类型 | 内容 | 放在哪里 | 用途 |
|---|---|---|---|
| 岗位画像数据 | 岗位名称、核心能力、典型任务、关键词、企业常见考察方向 | 公共岗位画像库 | 让面试题更贴近岗位 |
| 面试题库数据 | 岗位、知识点、问题、参考答题点、难度、评分点 | 岗位知识库 / CSV 样例 | 让面试官知道“这个岗位大致考什么” |
| 简历-JD 匹配数据 | 简历文本、岗位描述、匹配标签或分数 | 离线评测集 | 验证关键词、语义匹配和规则评分 |
| AI-人工评分数据 | 简历、岗位、问题、回答、AI 分、人工分、差值 | 后台评测样本 / JSONL | 做评分对比、校准和微调准备 |

不要把所有东西都叫“训练数据”。当前阶段更准确的说法是：

- 岗位知识库：用于 RAG/知识注入，不是直接训练模型。
- 评测数据集：用于验证系统效果。
- 标注数据集：用于后续评分校准和微调准备。

## 2. 怎么找数据

### 2.1 岗位画像数据

优先找公开、可引用、结构化程度高的资料：

- O*NET：可查职业任务、技能、工作活动，适合抽取岗位任务和能力词。
- ESCO：欧盟职业与技能分类，适合补充职业、技能、能力关系。
- BLS Skills Data：可作为职业技能重要度的参考。
- 招聘网站公开 JD：只建议人工摘录要点，不要批量爬取，不要复制整段企业 JD。
- 学校就业指导资料、公开岗位说明书：适合中文化改写。

### 2.2 面试题库数据

技术岗可以参考公开面试资料，但要注意许可和改写：

- 前端：JavaScript、浏览器、Vue/React、工程化、性能优化。
- 后端：语言基础、数据库、Redis、消息队列、并发、系统设计、线上排障。
- 测试：测试用例设计、接口测试、自动化、缺陷定位、性能测试。
- 算法/数据：机器学习基础、模型评估、SQL、指标体系、A/B 测试。

非技术岗主要来自岗位任务和场景：

- 产品：需求分析、竞品分析、PRD、优先级、上线复盘。
- 运营：活动执行、用户沟通、数据复盘、增长指标。
- HR/招聘：招聘流程、简历初筛、面试邀约、候选人沟通。
- 行政：会议组织、文档管理、流程执行、突发事务处理。

### 2.3 简历-JD 匹配数据

可以找公开的 synthetic 或匿名数据集做离线实验。注意：

- 合成数据适合演示算法流程，但不能代表真实招聘。
- 真实简历数据可能涉及隐私，不要随便收集或上传。
- 所有数据进入项目材料前都要记录来源、许可和用途。

### 2.4 本项目自己产生的数据

这是最适合参赛答辩的数据：

1. 你上传一份简历。
2. 选择岗位画像。
3. 系统生成问题。
4. 你回答。
5. 系统评分。
6. 人工再评分。
7. 记录差值和原因。

这类数据最能证明“系统确实跑过、评过、对比过”。

## 3. 怎么加入当前项目

### 3.1 加入岗位知识库

最稳方式是通过公共岗位画像种子脚本：

```powershell
cd D:\apps\ai-interview\ai-interview-backend
python -m app.scripts.seed_competition_knowledge_bases
```

当前脚本会写入：

- 岗位名称
- 岗位类别
- 核心能力
- 典型任务
- 关键词
- 面试八股/知识点
- 企业常见考察方向
- 样例面试题
- 面试考察重点
- 评分提示

这些内容会进入公共岗位画像库，并被切片服务拆成可用于出题和追问的知识片段。

### 3.2 加入面试题库样例

可以先维护 CSV：

```text
docs/competition/datasets/岗位面试题库样例.csv
```

字段建议：

```text
岗位,岗位类别,知识点,企业常见考察方向,样例问题,参考答题点,难度,数据来源,许可备注
```

等样例稳定后，再把高质量题目整理进公共岗位画像。

### 3.3 加入评测数据集

可以先维护 CSV：

```text
docs/competition/datasets/数据集清单模板.csv
```

字段建议：

```text
数据集名称,数据集类型,来源链接,许可,主要字段,项目用途,当前状态,备注
```

后续后台导出的 JSONL 用于微调准备：

```json
{
  "target_position": "Python后端开发工程师",
  "position_profile": "岗位画像摘要",
  "question": "面试问题",
  "answer": "候选人回答",
  "ai_score": 7.8,
  "human_score": 8.0,
  "rubric": "评分依据",
  "accepted": true
}
```

## 4. 数据集建设顺序

不要先追求“下载一个大数据集然后微调”。本项目更适合这个顺序：

1. 先做岗位知识库：让系统问得像目标岗位。
2. 再做小规模测试案例：5-10 个技术岗和非技术岗案例。
3. 再做人工评分对比：每个案例至少 2 人评分。
4. 再做规则校准：看 AI 分和人工分差在哪里。
5. 最后再准备 JSONL：数据够了才谈微调。

也就是说，`职启智评项目升级流程手册.md` 是主线；`技术深度升级路线.md` 是技术加深说明；本手册是阶段 3、4、10、11 的数据建设说明。

## 5. 验收标准

| 阶段 | 过关效果 |
|---|---|
| 岗位知识库 | 每个重点岗位都能说清“企业大致考什么” |
| 面试题库 | 每个重点岗位至少 5 个样例问题 |
| 简历-JD 匹配数据 | 至少 5 条样例可跑关键词覆盖和语义分 |
| AI-人工评分数据 | 至少 5 条样例有 AI 分、人工分、差值和原因 |
| 微调准备 | 有 JSONL 结构，但不宣称已真实微调 |
