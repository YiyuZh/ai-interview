# 数据与样本现状

## 1. 样本来源

当前微调准备样本分为两类：

| 类型 | 用途 | 是否可用于证明真实效果 |
|---|---|---|
| 真实授权样本 | 来自真实用户闭环案例，经过授权、去标识化和人工复核 | 可以，但数量和质量必须如实记录 |
| 构造样本 | 用于补齐格式、岗位覆盖和任务示例 | 不可以冒充真实用户样本 |

## 2. 真实样本进入训练前的门槛

真实样本必须满足：

- 用户完成本次案例数据贡献授权。
- 案例完成简历上传、能力诊断、简历润色、学习任务、至少 3 轮面试、报告和训练复盘中的关键链路。
- 后台人工评分或人工复核完成。
- 样本不含直接身份标识。
- 样本有明确岗位、能力缺口、证据状态和面试上下文。

## 3. 去标识化规则

必须删除或遮挡：

- 姓名。
- 手机号。
- 邮箱。
- 证件号。
- 学号。
- 详细住址。
- 文件名中的个人标识。

为保证岗位诊断质量，可以在明确告知和授权前提下保留：

- 学校名称或学校层级。
- 专业。
- 教育经历。
- 实习经历。
- 项目经历。
- 技能栈。
- 目标岗位。
- 面试问答。
- 报告建议。
- 后台人工评分。

注意：这属于去标识化或脱敏，不应宣称为完全匿名。

## 4. 最低训练门槛

创建 OpenAI SFT job 前，最低门槛：

```text
real_authorized_samples >= 3
train_samples >= 10
```

建议门槛：

```text
real_authorized_samples >= 10
train_samples = 30-50
validation_samples >= 5
```

如果不足，下一步不是降低标准，而是继续补样本。

## 5. 三岗位样本计划

C1/C2/C3 真实闭环已后置，但仍是补真实样本的优先来源：

| 案例 | 岗位 | 验证重点 |
|---|---|---|
| C1 | Python 后端开发工程师 | Redis、SQL、接口、并发、项目排障追问 |
| C2 | 产品助理 / 产品经理实习生 | 需求分析、PRD、产品指标、非技术岗表达 |
| C3 | 人力资源专员 | 招聘漏斗、候选人沟通、HR 面经、报告建议 |

每个案例至少应形成：

- 一份去标识化简历摘要。
- 一个目标岗位画像。
- 一组能力缺口。
- 至少 3 轮面试问答。
- 一份报告或复盘摘要。
- 一份后台人工评分。
- 一个样本授权状态。

## 6. 样本质量标准

高质量 SFT 样本应满足：

- 输入信息足够，但不过度暴露隐私。
- 输出追问围绕能力缺口。
- 输出不编造候选人经历。
- 输出要求候选人给出具体场景、行动、结果和指标。
- 输出结构稳定，方便自动评估。
- metadata 能追溯岗位、样本来源、授权状态和人工质量标签。
