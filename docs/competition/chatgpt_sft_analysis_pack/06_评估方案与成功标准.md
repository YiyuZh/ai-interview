# 评估方案与成功标准

## 1. Eval 的目的

Eval 不是为了证明模型“一定变强”，而是为了判断微调是否让模型更稳定地完成职启智评的业务任务。

第一版重点评估：

- 追问是否围绕能力缺口。
- 是否避免编造候选人经历。
- 是否要求具体场景、行动、结果和指标。
- 是否正确使用岗位画像和证据状态。
- 输出格式是否稳定。
- 是否比 base model 更适合进入报告和人工评分闭环。

## 2. 对比对象

至少比较两个模型：

| 对象 | 说明 |
|---|---|
| base model | OpenAI 当前支持 fine-tuning 的原始模型 |
| fine-tuned model | OpenAI SFT job 成功后得到的模型 |

## 3. Holdout 样本要求

最低要求：

```text
validation_samples >= 5
```

更建议：

```text
validation_samples >= 10
覆盖 Python 后端、产品助理、人力资源专员等不同岗位
```

Holdout 样本不能和训练样本重复。

## 4. 评估维度

建议每条样本按 1-5 分评分：

| 维度 | 高分标准 |
|---|---|
| 能力缺口聚焦 | 问题直接围绕缺口能力，不跑题 |
| 证据约束 | 不把岗位知识写成候选人经历 |
| 追问深度 | 要求候选人说明场景、任务、行动、结果、指标 |
| 岗位贴合 | 问法符合目标岗位真实工作内容 |
| 格式稳定 | 输出结构可解析，字段稳定 |
| 可用于报告 | 问题能帮助后续生成诊断报告和学习建议 |

## 5. 成功标准

第一版微调成功不要求给出夸张指标。建议成功标准为：

- OpenAI job 成功，拿到 `fine_tuned_model`。
- 保存 `job_record.json` 和 `job_status.json`。
- 训练样本统计可追溯，包括真实授权样本数、构造样本数、岗位覆盖。
- 至少 5 条 holdout 样本完成 base/fine-tuned 对比。
- eval 结果能指出优势、问题和下一轮样本改进方向。

## 6. 答辩可用表达

未创建 job 前：

```text
已完成 OpenAI SFT 数据准备与启动脚本，能够把授权、人工复核后的案例转换成 OpenAI chat fine-tuning JSONL。
```

job 已创建但未完成：

```text
已启动一次 OpenAI SFT 训练作业，当前保留 job id 和样本统计，等待训练状态完成。
```

job 成功后：

```text
已完成一次 OpenAI SFT 实验，并保存 job id、模型 id、样本统计和训练状态记录。
```

eval 完成后：

```text
已用 holdout 样本对比 base model 和 fine-tuned model，围绕追问聚焦、证据约束和格式稳定性进行评估。
```

## 7. 不应使用的表达

不要说：

- 自研底层大模型。
- 已经拥有大规模真实训练集。
- 构造样本来自真实用户。
- 未授权用户样本也进入训练。
- 没有 eval 结果时就宣称效果提升。
- 用单个样例截图代替整体评估。
