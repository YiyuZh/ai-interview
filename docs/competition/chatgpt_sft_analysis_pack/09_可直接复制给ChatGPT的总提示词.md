# 可直接复制给 ChatGPT 的总提示词

下面这段可以直接复制给 ChatGPT。复制时，把本文件夹里的其他 Markdown 内容一起提供给它。

```text
你是一名熟悉 OpenAI fine-tuning、AI 产品落地、数据合规和比赛答辩的技术顾问。

我有一个项目叫“职启智评”，定位是就业能力诊断与提升平台，不是单纯 AI 面试官。项目主链路是：

简历解析 -> 目标岗位 -> 岗位画像 / 岗位知识库 -> 简历证据评分 -> 能力差距诊断 -> 学习任务 -> AI 模拟面试 -> 报告生成 -> 训练复盘 -> 本次案例授权 -> 后台人工评分沉淀 -> 评测与微调样本准备。

我现在想推进 OpenAI SFT 真实微调，但必须基于真实项目状态，不要夸大。

当前事实：

1. 项目已经完成 OpenAI SFT 数据准备、JSONL 转换、创建 job、查询状态和 eval 对比的脚本设计。
2. 当前还不能直接宣称已经完成真实微调。只有 OpenAI fine-tuning job 成功并拿到 fine_tuned_model 后，才能说完成一次 OpenAI SFT 实验。
3. 第一版计划只做 OpenAI supervised fine-tuning，不做 RFT、DPO、LoRA、QLoRA 或本地开源模型训练。
4. 第一版建议训练任务是：根据去标识化简历摘要、目标岗位、能力缺口、证据状态、岗位画像切片和面试上下文，生成下一轮高质量面试追问。
5. 真实样本进入训练前必须满足：用户同意本次案例加入去标识化评测库、后台人工复核完成、去除直接身份标识。
6. 构造样本可以补格式和岗位覆盖，但必须标记 sample_origin=constructed，不能说成真实用户数据。
7. 当前最低训练门槛是：真实授权样本不少于 3 条，总训练样本不少于 10 条；更建议真实授权样本不少于 10 条，总样本 30-50 条，验证集不少于 5 条。
8. C1/C2/C3 三岗位真实闭环还没有作为最终结果完成，后续仍要补真实样本。
9. 本地 main 领先远端 origin/main 16 个提交，服务器开跑前需要先 push，否则服务器 git pull 拉不到最新脚本。
10. 不能说自研底层大模型，不能说已经拥有大规模真实训练数据，不能说构造样本来自真实用户，不能在没有 eval 结果时宣称效果提升。

请你基于我提供的资料包，完成以下分析：

1. 判断当前项目是否可以直接开始 OpenAI SFT 训练；如果不能，列出阻塞条件和解决顺序。
2. 评估“第一版只训练面试追问生成任务”是否合理；如果合理，请给出推荐输入字段、输出格式和样本结构。
3. 设计训练集、验证集、构造样本和真实授权样本的配比方案。
4. 设计一套轻量但可信的 eval，包括评分维度、评分表、通过标准和答辩呈现方式。
5. 给出 OpenAI SFT 从 dry-run 到创建 job 到查询状态再到 eval 的执行路线。
6. 给出比赛答辩口径：哪些可以说，哪些要谨慎说，哪些不能说。
7. 给出接下来 7 天的具体执行计划，每天写清目标、产出、命令或操作、验收标准。

要求：

- 不要建议使用未授权用户数据。
- 不要建议把构造样本冒充真实数据。
- 不要建议跳过 eval。
- 不要夸大当前成果。
- 回答要面向一个想尽快拿比赛成果、但又不想违法或被评委问倒的项目负责人。
```
