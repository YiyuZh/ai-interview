# 微调目标与边界

## 1. 为什么要做微调

当前项目已经可以调用通用大模型完成简历诊断、面试追问和报告生成，但通用模型存在几个问题：

- 容易泛泛地问面试题，和岗位能力缺口不够贴合。
- 容易把岗位知识库里的要求误写成候选人的真实经历。
- 输出格式可能不稳定，不利于报告、评分和后台沉淀。
- 对不同岗位的追问颗粒度不一致。
- 很难稳定体现“证据状态”，例如已证明、间接证明、仅声明、缺失。

微调的目标不是训练一个底层大模型，而是让模型更稳定地执行职启智评的业务任务。

## 2. 第一版 SFT 目标

第一版建议只做 OpenAI supervised fine-tuning，聚焦一个窄任务：

```text
根据去标识化简历摘要、目标岗位、能力缺口、证据状态、岗位画像切片和面试上下文，生成下一轮高质量面试追问。
```

这个任务适合作为第一版原因：

- 输入输出结构清晰。
- 与岗位画像和 RAG 链路强相关。
- 可以通过人工评分判断追问质量。
- 更容易用构造样本补齐格式。
- 可以在答辩中解释为“让模型更贴合就业诊断场景”。

## 3. 不建议第一版训练的内容

第一版不建议把所有任务混在一起训练，例如：

- 简历解析。
- 简历润色全文生成。
- 完整报告生成。
- 学习任务生成。
- 多轮对话全流程。
- 候选人综合评分。

原因是这些任务输出形态差异很大，样本量少时容易互相干扰，也不容易评估微调是否真的有效。

## 4. 技术边界

第一版只做：

- OpenAI SFT。
- chat fine-tuning JSONL。
- 授权真实样本 + 构造样本。
- 固定 holdout 对比评估。

第一版不做：

- RFT。
- DPO。
- LoRA / QLoRA。
- 本地开源模型训练。
- 多模型路由训练平台。
- 面向普通后台管理员的训练 UI。

## 5. 答辩边界

可以说：

```text
我们已经搭建了从用户授权、去标识化、人工评分到 OpenAI SFT 数据准备的闭环，并设计了训练 job 与 eval 对比流程。
```

拿到 `fine_tuned_model` 之后可以说：

```text
我们完成了一次 OpenAI SFT 实验，并保存了 job id、模型 id、样本统计和评估记录。
```

不能说：

```text
我们自研了底层大模型。
我们已经拥有大规模真实训练数据。
构造样本来自真实用户。
未授权用户数据也用于训练。
微调一定带来确定比例的效果提升。
```
