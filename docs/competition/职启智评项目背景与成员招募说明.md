# 职启智评项目背景与五人维护分工说明

适用对象：当前五位项目维护成员、后续交接成员和需要快速了解项目分工的协作者。

当前版本：2026-05-10，根据 `职启智评项目升级流程手册.md` 和 `任务记录文档.md` 整理。

## 1. 一句话介绍

职启智评是一个面向大学生求职训练的就业能力诊断与提升平台，核心闭环是：简历解析 -> 岗位能力图谱 -> 能力差距诊断 -> 学习任务 -> 模拟面试 -> 训练复盘。

它不是只做“AI 随机问面试题”的工具，而是希望把学生的简历证据、目标岗位要求、岗位知识库、学习路线和训练记录连起来，形成持续提升的求职能力系统。

## 2. 项目背景

项目最早来自一个 AI 模拟面试系统，能够让用户上传简历、选择岗位、进行模拟面试并查看报告。初赛后根据评委反馈和项目复盘，主线已经从“模拟面试工具”升级为“就业能力诊断与提升平台”。

现在的产品逻辑是：

1. 用户上传简历，系统提取教育经历、项目经历、技能、实习经历和可验证证据。
2. 用户选择目标岗位，系统读取岗位画像和岗位能力模型。
3. 系统对比“简历已有证据”和“岗位要求能力”，生成能力差距。
4. 系统把能力差距转成学习任务和学习路线。
5. 用户围绕当前岗位进入模拟面试，系统根据简历证据、岗位知识切片和面试问答经验生成追问。
6. 面试后生成报告，并进入训练复盘，记录下一轮提升目标。
7. 后台持续维护岗位画像、岗位知识库、学习路线、面试问答经验和测评样本。

项目名称继续使用“职启智评”，工程目录、GitHub 仓库和服务器路径继续保留 `ai-interview`，避免破坏现有部署链路。

## 3. 技术流程

### 3.1 用户端流程

用户端主要面向学生和求职训练用户，包含：

- 简历上传与分析。
- 目标岗位选择。
- 能力诊断页面。
- 学习任务页面。
- 模拟面试页面。
- 面试报告页面。
- 训练复盘页面。
- 私有岗位画像与问答经验维护。

用户端技术重点是 Vue 页面交互、表单体验、报告可读性、本地与服务器数据同步，以及在出错时给出清楚提示。

### 3.2 后台端流程

后台端主要面向项目维护者和运营成员，包含：

- 公共岗位画像库维护。
- 岗位知识库四分区维护：岗位要求、问答经验、能力模型、面试追问。
- 结构化面试问答经验维护。
- 学习路线管理。
- 学习路线质量审核、批量导入导出、复制、任务预览。
- 学习路线覆盖矩阵。
- 用户、面试记录、测评样本和人工评分对比管理。

后台端的目标不是“展示给评委看的静态页”，而是让项目后续真的可以维护岗位知识资产、学习路线资产和测评样本。

### 3.3 后端与数据库流程

后端主要使用 FastAPI、SQLAlchemy、Alembic、PostgreSQL、Redis、Celery 和 Docker。核心数据包括：

- 用户与登录状态。
- 简历与分析结果。
- 岗位画像与岗位知识库。
- 岗位知识切片。
- 面试记录与消息。
- 面试报告。
- 学习任务。
- 训练复盘。
- 学习路线阶段。

当前策略是尽量复用已有字段和接口；只有学习任务、训练复盘、学习路线维护这类必须账号级持久化的数据，才新增数据库迁移。

### 3.4 AI 与能力诊断流程

AI 能力主要来自大语言模型 API、结构化 Prompt、证据约束和规则评分组合。项目不宣称自研大模型，也不把真实微调写成已经上线的能力。

当前能力包括：

- 简历结构化解析。
- 岗位匹配分析。
- 关键词覆盖与规则评分。
- 岗位能力差距计算。
- 基于能力差距生成学习任务。
- 基于简历证据和知识切片生成面试追问。
- 报告与训练复盘内容组织。

系统强调“证据约束”：简历没有写的能力不能直接判断为已掌握，只能转成追问或学习任务。

### 3.5 岗位知识库与切片流程

岗位画像不是单一文本，而是逐步升级为“岗位能力图谱 + 岗位知识库”。知识库分为四类：

- 岗位要求：硬性要求、加分项、公司侧重点。
- 问答经验：真实面试问题、参考回答要点、复盘反思、来源审核状态。
- 能力模型：能力项、等级标准、权重、可提升系数。
- 面试追问：证据追问规则、评分关注点。

后端会把岗位知识库切成可检索的知识切片。结构化问答经验会按审核状态参与或不参与面试追问：可入库正常参与，仅参考降低优先级，待核验和不采用不进入启用切片。

## 5. 项目优势

1. 场景真实：围绕大学生求职训练，而不是泛泛聊天或单次问答。
2. 多岗位生态：技术岗和非技术岗都能覆盖，后续可以继续扩展。
3. 证据约束清晰：简历里有证据才判断掌握，没有证据就转成追问或学习任务。
4. 知识库可维护：岗位要求、问答经验、能力模型和追问规则都可以逐步沉淀。
5. 学习提升闭环完整：不是只指出问题，还能生成学习任务、记录进度、进入复盘。
6. 后台可运营：岗位画像、学习路线、问答经验、测评样本都可以由成员持续维护。
7. 可沉淀数据集：后续可以形成简历证据、岗位能力、面试问答、人工评分和 AI 输出对比数据。
8. 适合分工协作：开发、产品、数据标注、资料搜集、学习路线、PPT 演示都能并行推进。

## 6. 五人维护分工

项目已经确定由五个人长期维护。后续不再按“招募很多岗位”的方式分散管理，而是把相似方向合并成五个明确位置。每个人都需要有固定交付物，不能只提出想法。

### 6.1 角色 1：技术全栈与部署负责人

负责范围：

- 前端、后端、数据库、服务器、GitHub、Docker、Redis、Celery、接口、日志和构建全部归这个角色统筹。
- 负责用户端、后台端、后端 API、数据库迁移和线上部署链路。
- 负责判断技术问题是代码错误、数据库迁移问题、模型配置问题，还是服务器环境问题。

主要任务：

- 执行服务器统一验收：拉取最新代码、重建服务、执行 Alembic、执行岗位画像种子脚本、跑验证脚本。
- 修复真实运行中的第一条后端异常，尤其是缺表、缺字段、接口 500、开始面试失败、测评样本错误等问题。
- 维护用户端页面、后台页面和后端接口的基本可用性。
- 保证学习任务、训练复盘、学习路线、案例标注等账号级数据能正常保存和读取。
- 控制 Git 提交范围，不使用 `git add .` 混入构建产物或无关资料。

可交付物：

- 可部署代码和通过构建的前后台页面。
- Alembic 迁移执行结果。
- 服务器验证报告。
- 错误日志定位记录。
- Git 提交和部署记录。

### 6.2 角色 2：产品流程与验收负责人

负责范围：

- 负责把项目主线串起来：简历解析 -> 能力诊断 -> 学习任务 -> 模拟面试 -> 报告 -> 训练复盘。
- 负责检查页面是否真的能服务用户，而不是只堆功能入口。
- 负责把每次验收结果写成清晰的通过、不通过、待修复记录。

主要任务：

- 设计并维护主流程验收清单。
- 用 R1-R5 真实案例跑完整闭环，记录每一步是否成功。
- 检查用户端首页、能力诊断页、学习任务页、报告页、训练复盘页是否职责清楚。
- 把失败页面、失败接口、异常提示截图或日志归档给技术负责人。
- 防止项目继续出现重复入口、无效页面和未验收功能被写成已完成。

可交付物：

- 主流程验收清单。
- R1-R5 闭环跑测记录。
- 页面问题清单。
- 通过/不通过结论。
- 下一轮修复优先级。

### 6.3 角色 3：岗位知识库与面经负责人

负责范围：

- 负责岗位画像、岗位要求、公司侧重点、结构化面试问答经验和知识切片内容质量。
- 负责技术岗和非技术岗的知识库内容补全，不负责写代码。
- 负责判断资料能不能进入公共知识库。

主要任务：

- 补全 12 个岗位的硬性要求、加分项、常见工作任务和公司侧重点。
- 搜集公开面经、公开岗位要求、招聘 JD 和开源面试资料，并做来源、时效、许可和可信度判断。
- 把资料整理为结构化问答经验：真实问题、参考回答要点、复盘反思、公司/场景、考察能力、难度、来源说明、审核状态。
- 标注资料状态：可入库、仅参考、待核验、不采用。
- 与技术负责人配合检查重建切片后，问答经验是否按审核状态参与面试追问。

可交付物：

- 岗位要求整理稿。
- 公司侧重点清单。
- 结构化问答经验卡片。
- 来源审核表。
- 知识库补全记录。

### 6.4 角色 4：数据标注与人工评分负责人

负责范围：

- 负责真实案例、简历证据、AI 输出核查、人工评分、重复运行和数据集沉淀。
- 负责让项目有“可验证数据”，而不是只有演示页面。

主要任务：

- 维护 R1-R5 真实或公开授权简历案例。
- 标注简历中能支持岗位能力的证据，区分简历原文、岗位要求、AI 推断和人工判断。
- 在后台“案例标注”中填写 `case_id`、人工总分、证据对齐分、问题质量分、报告可执行性分、学习任务可执行性分和数据集归档字段。
- 对比 AI 评分和人工评分，记录差异原因。
- 对关键案例做 3 次重复运行，记录评分波动和不稳定点。

可交付物：

- 真实闭环验收记录。
- 简历证据标注表。
- 人工评分表。
- AI 与人工评分差异记录。
- 数据集清单和重复运行记录。

### 6.5 角色 5：学习路线与答辩材料负责人

负责范围：

- 负责学习任务质量、学习路线阶段、答辩材料、讲稿、项目说明和展示故事线。
- 负责把真实验收结果转成能讲清楚的项目材料。

主要任务：

- 抽查系统生成的学习任务是否具体、可执行、有练习、有产出物、有验收方式。
- 维护 Python、Java、前端、测试、产品、运营、HR 等岗位的学习路线阶段。
- 检查非技术岗任务不能写成不合适的小 Demo，技术岗任务要有项目练习和排障复盘。
- 根据真实案例更新 PPT、讲稿、项目说明书、答辩问答和演示流程。
- 确保材料不夸大能力，不写自研大模型、真实微调已上线或商业落地。

可交付物：

- 学习路线阶段表。
- 学习任务质量检查表。
- PPT 和逐页讲稿。
- 项目说明与答辩问答。
- 演示流程检查表。

## 7. 五人协作原则

1. 技术问题统一交给技术全栈与部署负责人，不再拆成前端、后端、运维多个零散入口。
2. 产品流程与验收负责人决定“这条流程是否真的跑通”，不是看页面有没有做出来。
3. 岗位知识库与面经负责人只负责可审核资料和内容质量，不把来源不明材料直接入库。
4. 数据标注与人工评分负责人只记录真实结果，不虚构分数，不把待跑测写成成功。
5. 学习路线与答辩材料负责人必须等真实验收结果出来后再更新材料，不能提前夸大。
6. 所有人都要遵守阶段化记录：做了什么、验证了什么、没完成什么，都要写清楚。

## 9. 五人维护协作方式

1. 每周先看 `职启智评项目升级流程手册.md` 的“一页总览”，确认当前最高优先级。
2. 每个人只认领自己角色下的任务，跨角色任务必须明确主负责人。
3. 每个任务必须有可检查交付物，例如验收记录、结构化面经、人工评分表、学习路线表、PPT 页或错误日志。
4. 功能没有在服务器跑通前，只能写“本地初版”或“待服务器验收”，不能写“已完成上线”。
5. AI 输出不能直接当结论，必须由数据标注与人工评分负责人或对应内容负责人复核。
6. 项目材料必须跟随真实系统状态更新，不能把未实现、未验收、未跑通的能力写进正式展示。
7. 每次阶段性调整后，同步更新 `任务记录文档.md`；涉及主流程变化时，同步更新升级流程手册。

项目现在最需要的不是继续堆概念，而是五个人把各自负责的证据、内容、验收、技术和材料稳定维护起来。每个人能把自己的交付物做扎实，这个项目才会像一个真正能长期运行的系统。
