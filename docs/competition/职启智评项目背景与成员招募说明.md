# 职启智评项目背景与五人维护分工说明

适用对象：当前五位项目维护成员、后续交接成员和需要快速了解项目分工的协作者。

当前版本：2026-05-10，根据 `职启智评项目升级流程手册.md` 和 `任务记录文档.md` 整理。

## 1. 一句话介绍

职启智评是一个面向大学生求职训练的就业能力诊断与提升平台，核心闭环是：简历解析 -> 岗位能力图谱 -> 能力差距诊断 -> 学习任务 -> 模拟面试 -> 训练复盘。

它不是只做“AI 随机问面试题”的工具，而是希望把学生的简历证据、目标岗位要求、岗位知识库、学习路线和训练记录连起来，形成持续提升的求职能力系统。

## 2. 项目背景

项目最早来自一个 AI 模拟面试系统，能够让用户上传简历、选择岗位、进行模拟面试并查看报告。初赛后根据评委反馈和项目复盘，主线已经从“模拟面试工具”升级为“就业能力诊断与提升平台”。

现在的产品逻辑是：

1. 用户上传简历，系统提取教育经历、项目经历、技能、实习经历和可验证证据。
2. 用户选择目标岗位，系统读取岗位画像和岗位能力模型。
3. 系统对比“简历已有证据”和“岗位要求能力”，生成能力差距。
4. 系统把能力差距转成学习任务和学习路线。
5. 用户围绕当前岗位进入模拟面试，系统根据简历证据、岗位知识切片和面试问答经验生成追问。
6. 面试后生成报告，并进入训练复盘，记录下一轮提升目标。
7. 后台持续维护岗位画像、岗位知识库、学习路线、面试问答经验和测评样本。

项目名称继续使用“职启智评”，工程目录、GitHub 仓库和服务器路径继续保留 `ai-interview`，避免破坏现有部署链路。

## 3. 技术流程

### 3.1 用户端流程

用户端主要面向学生和求职训练用户，包含：

- 简历上传与分析。
- 目标岗位选择。
- 能力诊断页面。
- 学习任务页面。
- 模拟面试页面。
- 面试报告页面。
- 训练复盘页面。
- 私有岗位画像与问答经验维护。

用户端技术重点是 Vue 页面交互、表单体验、报告可读性、本地与服务器数据同步，以及在出错时给出清楚提示。

### 3.2 后台端流程

后台端主要面向项目维护者和运营成员，包含：

- 公共岗位画像库维护。
- 岗位知识库四分区维护：岗位要求、问答经验、能力模型、面试追问。
- 结构化面试问答经验维护。
- 学习路线管理。
- 学习路线质量审核、批量导入导出、复制、任务预览。
- 学习路线覆盖矩阵。
- 用户、面试记录、测评样本和人工评分对比管理。

后台端的目标不是“展示给评委看的静态页”，而是让项目后续真的可以维护岗位知识资产、学习路线资产和测评样本。

### 3.3 后端与数据库流程

后端主要使用 FastAPI、SQLAlchemy、Alembic、PostgreSQL、Redis、Celery 和 Docker。核心数据包括：

- 用户与登录状态。
- 简历与分析结果。
- 岗位画像与岗位知识库。
- 岗位知识切片。
- 面试记录与消息。
- 面试报告。
- 学习任务。
- 训练复盘。
- 学习路线阶段。

当前策略是尽量复用已有字段和接口；只有学习任务、训练复盘、学习路线维护这类必须账号级持久化的数据，才新增数据库迁移。

### 3.4 AI 与能力诊断流程

AI 能力主要来自大语言模型 API、结构化 Prompt、证据约束和规则评分组合。项目不宣称自研大模型，也不把真实微调写成已经上线的能力。

当前能力包括：

- 简历结构化解析。
- 岗位匹配分析。
- 关键词覆盖与规则评分。
- 岗位能力差距计算。
- 基于能力差距生成学习任务。
- 基于简历证据和知识切片生成面试追问。
- 报告与训练复盘内容组织。

系统强调“证据约束”：简历没有写的能力不能直接判断为已掌握，只能转成追问或学习任务。

### 3.5 岗位知识库与切片流程

岗位画像不是单一文本，而是逐步升级为“岗位能力图谱 + 岗位知识库”。知识库分为四类：

- 岗位要求：硬性要求、加分项、公司侧重点。
- 问答经验：真实面试问题、参考回答要点、复盘反思、来源审核状态。
- 能力模型：能力项、等级标准、权重、可提升系数。
- 面试追问：证据追问规则、评分关注点。

后端会把岗位知识库切成可检索的知识切片。结构化问答经验会按审核状态参与或不参与面试追问：可入库正常参与，仅参考降低优先级，待核验和不采用不进入启用切片。

## 5. 项目优势

1. 场景真实：围绕大学生求职训练，而不是泛泛聊天或单次问答。
2. 多岗位生态：技术岗和非技术岗都能覆盖，后续可以继续扩展。
3. 证据约束清晰：简历里有证据才判断掌握，没有证据就转成追问或学习任务。
4. 知识库可维护：岗位要求、问答经验、能力模型和追问规则都可以逐步沉淀。
5. 学习提升闭环完整：不是只指出问题，还能生成学习任务、记录进度、进入复盘。
6. 后台可运营：岗位画像、学习路线、问答经验、测评样本都可以由成员持续维护。
7. 可沉淀数据集：后续可以形成简历证据、岗位能力、面试问答、人工评分和 AI 输出对比数据。
8. 适合分工协作：开发、产品、数据标注、资料搜集、学习路线、PPT 演示都能并行推进。

## 6. 五人维护分工

项目已经确定由五个人长期维护。后续不再按“前端、后端、产品、运营、标注、资料、PPT”等零散岗位拆分，而是按真实工作量重新合并。技术与最终运行验收由一个人统一负责；岗位资料搜集拆成技术岗和非技术岗两个人，避免一个人承担全部岗位资料；数据标注写成明确表格和字段任务。

### 6.1 角色 1：技术全栈与验收负责人

负责范围：

- 前端、后端、数据库、服务器、GitHub、Docker、Redis、Celery、接口、日志、构建和部署全部由这个角色统筹。
- 同时负责最终运行验收：页面有没有跑通、接口有没有报错、服务器有没有完成迁移，都由这个角色给出最终技术结论。
- 不再单独设置轻量验收角色，避免验收职责过轻、问题归属不清。

主要任务：

- 执行服务器统一验收：拉取最新代码、重建服务、执行 Alembic、执行岗位画像种子脚本、跑验证脚本。
- 跑通主流程：上传简历、能力诊断、学习任务、模拟面试、报告、训练复盘、后台案例标注。
- 修复真实运行中的第一条技术错误，尤其是缺表、缺字段、接口 500、开始面试失败、测评样本错误等问题。
- 维护用户端、后台端和后端接口的基本可用性。
- 控制 Git 提交范围，不使用 `git add .` 混入构建产物或无关资料。

可交付物：

- 服务器验证报告。
- 主流程跑测记录。
- 可部署代码和构建结果。
- Alembic 迁移执行结果。
- 错误日志定位记录。
- Git 提交和部署记录。

### 6.2 角色 2：技术岗岗位资料与面经负责人

负责范围：

- 负责技术岗资料，不再承担非技术岗资料。
- 覆盖 Python 后端、Java 后端、前端开发、测试工程师、算法工程师、数据分析师等岗位。
- 负责这些岗位的岗位要求、企业侧重点、面试题方向、参考回答要点和来源审核。

主要任务：

- 为技术岗整理硬性要求、加分项、常见工作任务和公司侧重点。
- 搜集公开面经、招聘 JD、开源面试资料和技术面试资料，并记录来源、年份、许可和可信度。
- 把资料改写为结构化问答经验：真实问题、参考回答要点、复盘反思、公司/场景、考察能力、难度、来源说明、审核状态。
- 技术岗内容要覆盖语言基础、数据库、Redis/缓存、HTTP/网络、并发/异步、系统设计、项目追问、线上排障等方向。
- 标注资料状态：可入库、仅参考、待核验、不采用。

可交付物：

- 技术岗岗位要求整理稿。
- 技术岗面经候选清单。
- 技术岗结构化问答经验卡片。
- 技术资料来源审核表。
- 技术岗知识库补全记录。

### 6.3 角色 3：非技术岗岗位资料与面经负责人

负责范围：

- 负责非技术岗资料，不再承担技术岗资料。
- 覆盖产品助理、运营助理、新媒体运营、人力资源专员、招聘助理、行政管理助理等岗位。
- 负责这些岗位的岗位要求、场景任务、企业侧重点、面试问题、复盘反思和来源审核。

主要任务：

- 为非技术岗整理硬性要求、加分项、常见任务、工作场景和公司侧重点。
- 搜集公开面经、招聘 JD、职业能力资料和岗位经验贴，并记录来源、年份、许可和可信度。
- 把资料改写为结构化问答经验：真实问题、参考回答要点、复盘反思、公司/场景、考察能力、难度、来源说明、审核状态。
- 非技术岗内容要覆盖沟通协作、流程意识、数据复盘、活动执行、需求分析、候选人沟通、文档规范等方向。
- 标注资料状态：可入库、仅参考、待核验、不采用。

可交付物：

- 非技术岗岗位要求整理稿。
- 非技术岗面经候选清单。
- 非技术岗结构化问答经验卡片。
- 非技术资料来源审核表。
- 非技术岗知识库补全记录。

### 6.4 角色 4：数据标注与人工评分负责人

负责范围：

- 负责把系统跑出来的真实案例变成可验证数据。
- 不只是“标注数据”四个字，而是要具体记录案例、证据、能力项、评分、差异原因和重复运行结果。
- 负责维护 R1-R5 案例台账、真实闭环 CSV、人工评分表和数据集清单。

主要任务：

- 为每个案例建立编号：R1、R2、R3、R4、R5。
- 记录每个案例的简历来源、目标岗位、跑测日期、模型、环境、系统版本和服务器提交号。
- 从简历中摘录能支持岗位能力的原文证据，不自己脑补经历。
- 将简历证据映射到能力项，例如沟通表达、数据分析、项目经历、岗位知识、工具使用、执行复盘、学习潜力等。
- 在后台“案例标注”中填写 `case_id`、`resume_source`、`human_overall_score`、`evidence_alignment_score`、`question_quality_score`、`report_actionability_score`、`learning_task_actionability_score`、`human_score_notes`、`dataset_split`。
- 对比 AI 分和人工分，写出差异原因：是证据不足、问题质量差、报告建议空泛，还是人工评分标准不一致。
- 至少选 1 个案例重复运行 3 次，记录评分波动和不稳定点。
- 维护 `真实闭环验收记录模板.csv`、人工评分表、数据集清单和失败记录。

可交付物：

- R1-R5 案例台账。
- 简历证据摘录表。
- 能力项映射表。
- 后台案例标注记录。
- AI 与人工评分对比表。
- 重复运行记录。
- 数据集清单和失败记录。

### 6.5 角色 5：学习路线与答辩材料负责人

负责范围：

- 负责学习任务质量、学习路线阶段、答辩材料、讲稿、项目说明和展示故事线。
- 这个角色不负责大规模搜集岗位面经，只根据岗位资料负责人和数据标注负责人提供的结果，整理学习路线与展示材料。
- 材料必须基于已验收结果，不提前夸大。

主要任务：

- 抽查系统生成的学习任务是否具体、可执行、有练习、有产出物、有验收方式。
- 维护不同岗位的学习路线阶段，把能力缺口拆成可执行学习任务。
- 检查非技术岗任务不能写成不合适的小 Demo，技术岗任务要有项目练习和排障复盘。
- 根据 R1-R5 真实案例更新 PPT、讲稿、项目说明书、答辩问答和演示流程。
- 确保材料不夸大能力，不写自研大模型、真实微调已上线或商业落地。

可交付物：

- 学习路线阶段表。
- 学习任务质量检查表。
- PPT 和逐页讲稿。
- 项目说明与答辩问答。
- 演示流程检查表。

## 7. 五人协作原则

1. 技术全栈与验收负责人负责最终运行结论；服务器、页面、接口是否可用，以他的验收记录为准。
2. 技术岗资料由角色 2 负责，非技术岗资料由角色 3 负责，两个资料负责人都要记录来源、时效和审核状态。
3. 数据标注与人工评分负责人只记录真实跑测和人工判断，不虚构分数，不把待跑测写成成功。
4. 学习路线与答辩材料负责人只能基于已验收结果写材料，不能提前把本地初版写成线上完成。
5. 跨角色任务必须明确主负责人。例如 R1 闭环由技术负责人跑通，数据负责人标注，材料负责人再整理成展示内容。
6. 所有人都要遵守阶段化记录：做了什么、验证了什么、没完成什么，都要写清楚。

| 9 | 根据真实案例更新 PPT、讲稿、项目说明和答辩问答 | 角色 5：学习路线与答辩材料负责人 |

## 9. 五人维护协作方式

1. 每周先看 `职启智评项目升级流程手册.md` 的“一页总览”，确认当前最高优先级。
2. 角色 1 先保证系统能跑；系统没跑通前，其他角色不把功能写成已完成。
3. 角色 2 和角色 3 每次提交资料时都要写来源链接、年份、审核状态和是否建议入库。
4. 角色 4 每次跑测后都要补案例编号、简历证据、能力项映射、人工评分和差异原因。
5. 角色 5 只根据真实验收和真实案例更新材料，不用未验证功能做宣传。
6. 每个任务必须有可检查交付物，例如验收记录、结构化面经、人工评分表、学习路线表、PPT 页或错误日志。
7. 每次阶段性调整后，同步更新 `任务记录文档.md`；涉及主流程变化时，同步更新升级流程手册。

项目现在最需要的不是继续堆概念，而是五个人把各自负责的技术、资料、标注、学习路线和材料稳定维护起来。每个人能把自己的交付物做扎实，这个项目才会像一个真正能长期运行的系统。
