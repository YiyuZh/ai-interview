# 三岗位真实闭环验收执行包

本文件用于执行阶段 129/132 的三岗位真实闭环验证。它不是功能说明，也不是演示稿，而是跑测时照填的执行清单。

## 1. 验收目标

阶段 128 已完成成员知识库服务器正式入库、usage 检查和 SQL 核验。阶段 129 已完成路线重构，阶段 132 要按本执行包验证这些补充资料是否真的被系统使用。

本轮只验证三类代表岗位：

| 验证任务 | CSV 案例编号 | 目标岗位 | 验证价值 |
|---|---|---|---|
| C1 | R2 | Python后端开发工程师 | 验证技术岗成员资料是否进入 RAG、面试追问、简历润色和能力诊断 |
| C2 | R1 | 产品助理/产品经理实习生 | 验证非技术岗岗位要求、场景题、学习任务和报告建议是否可用 |
| C3 | R3 | 人力资源专员 | 验证 HR 面经、岗位要求、招聘漏斗、候选人沟通和企业侧重点是否支撑面试与报告 |

说明：C1/C2/C3 是阶段 129 的验证任务名；R1/R2/R3 是真实闭环 CSV 中的案例编号。后续填写 CSV 时继续使用 R1/R2/R3，避免重复编号。

## 3. 单案例执行步骤

每个案例都按同一流程执行。

1. 登录用户端。
2. 上传对应简历 PDF 或使用对应案例简历。
3. 选择目标岗位。
4. 等待简历解析和能力诊断完成。
5. 检查能力诊断是否显示：岗位缺口、证据状态、待验证能力。
6. 进入简历润色，检查润色建议是否引用岗位要求且不编造经历；同时记录“本次参考的岗位资料”是否优先显示标准岗位画像，并能使用已归并的成员补充内容。
7. 将至少 1 个缺口能力加入学习任务。
8. 进入学习任务页，确认任务刷新后仍存在。
9. 开始模拟面试，至少完成 3 轮问答，推荐 5 轮。
10. 记录每轮问题的验证能力和问题来源。
11. 生成面试报告。
12. 进入训练复盘，保存复盘笔记和下一轮目标。
13. 登录后台，进入案例标注，填写人工评分和备注。
14. 将结果追加到 `docs/competition/datasets/真实闭环验收记录模板.csv`。

## 4. C1：Python 后端开发工程师

### 重点检查

- 是否检索到 Python 后端标准岗位画像中的成员补充面经或相关技术岗补充切片。
- 面试是否围绕接口、数据库、缓存、并发、项目排障追问。
- 如果简历只写“熟悉 Redis / SQL”，是否被标记为待验证，而不是直接高分。
- 简历润色是否只提示补充项目证据，不虚构 Redis、FastAPI、数据库项目。

## 8. 人工评分口径

后台案例标注至少填写：

| 字段 | 建议分值范围 | 评分依据 |
|---|---:|---|
| `human_overall_score` | 0-100 | 整体岗位适配度 |
| `evidence_alignment_score` | 0-100 | 报告是否严格基于简历证据和问答表现 |
| `question_quality_score` | 0-100 | 面试问题是否覆盖岗位关键能力并能验证掌握程度 |
| `report_actionability_score` | 0-100 | 报告建议是否具体、可执行 |
| `learning_task_actionability_score` | 0-100 | 学习任务是否有材料、练习、产出物和验收方式 |

评分后必须写 `human_score_notes`，说明 AI 判断与人工判断一致或不一致的原因。

## 9. 失败处理

如果任意案例失败，不继续跑下一个案例，先抓第一条错误。

后端日志：

```bash
docker compose logs --tail=300 app | grep -Ei "Traceback|UndefinedColumn|relation .* does not exist|interviews/start|learning-tasks|learning-plans|resume.*polish|position_knowledge|RAG|slice"
```

前端页面失败时记录：

- 页面路径。
- 操作步骤。
- 浏览器报错截图。
- 后端第一条异常。
- 失败后是否影响已保存任务或报告。
