### 0.1 当前唯一主线

阶段 126/127/128 已经把第一批成员资料从协作目录迁移到后端专用数据区，并在服务器完成正式导入、usage 检查和 SQL 核验。但该阶段采用 `成员资料补充：{岗位}` 独立公共画像作为低风险临时方案，容易造成同一岗位出现多份画像。阶段 134 将其纠偏为“标准岗位画像 + 成员补充层 + 面试经验专门管理视图”。

阶段 133 调整后的执行顺序固定为：

1. 语义修复：把已导入的 `成员资料补充：{岗位}` 合并回对应标准岗位画像，并停用历史补充画像。
2. 面试经验管理：后台增加按岗位查看问答经验的独立入口，仍复用 `## 问答经验` 分区和切片体系。
3. 功能自动化收口：运行核心功能自检，确认评价快照、润色、学习计划、AI 面试验证目标和岗位知识库同口径可用。
4. 三岗位真实闭环：按 C1/C2/C3 跑通能力诊断、简历润色、AI 面试、报告、训练复盘。
5. 人工评分沉淀：后台案例标注填写人工分、证据对齐、问题质量、报告可执行性和学习任务可执行性。
6. 缺口岗位补资料：只针对已确认缺口补测试工程师、数据分析师、新媒体运营、招聘助理、行政管理助理。
7. 材料同步：真实闭环有结果后，再更新申报书、答辩材料、项目说明和成员招募说明。

如果阶段 134 归并失败，只修第一条真实错误；如果核心功能自检失败，只修第一条失败；如果三岗位闭环失败，只修第一条真实错误；不要新增功能绕开验证。

### 0.2 阶段 134 语义修复路线

| 模块 | 修复目标 | 通过标准 |
|---|---|---|
| 成员资料导入 | 默认归并到已有标准岗位画像，不再默认新建 `成员资料补充：{岗位}` 并列画像 | Python 后端等已有岗位导入后仍只有标准岗位画像作为主要来源 |
| 历史补充画像 | 已存在的 `成员资料补充：{岗位}` 先备份、再合并、再停用 | 数据不丢失，后台仍可追溯历史补充资料 |
| 问答经验 | 面经作为岗位画像内的一等内容管理 | 后台可按岗位查看问题、回答要点、复盘、场景、能力、来源和审核状态 |
| RAG/润色来源 | 不再通过标题 `成员资料补充` 提高优先级 | 简历润色优先显示标准岗位画像，同时可使用合并后的成员面经和岗位要求 |
| 切片 | 合并后重建切片 | `source_section=interview_experience` 切片数量不减少，`不采用` 不启用 |

阶段 134 完成后，后续成员资料仍先进入待检查和数据包预检；正式入库时默认归并到标准岗位画像。只有为了回滚、对比或审计时，才允许显式使用 legacy/separate 模式生成独立补充画像。

### 0.3 阶段 129/132 人工执行路线

### 0.7 无效升级识别规则

- 如果阶段 133 自检没有 PASS，不继续人工跑大量案例或新增功能。
- 如果阶段 138 readiness 没有通过，不继续人工跑大量案例。
- 阶段 139 是已确认的小功能例外；完成后如果 C1/C2/C3 没有跑完并通过 `validate_stage138_closed_loop.py`，不继续新增页面、表、接口或模型策略。
- 如果成员资料没有被 RAG 或润色引用，先修检索/上下文接入，不继续加资料。
- 如果同一能力在诊断、润色、面试、学习任务中口径冲突，先收口阶段 124 的评价快照。
- 如果 AI 面试把岗位知识库当作候选人经历，先修阶段 117 的证据约束。
- 如果学习任务空泛，先在后台路线管理修任务质量，不继续扩展更多岗位。
- 如果只是材料表述问题，等真实闭环结果出来后再统一改材料。

### 0.8 成员资料收集区

团队资料统一放在 `docs/competition/成员资料/`。该目录用于收集五位维护成员的岗位资料、面经、案例标注、学习路线、答辩材料和服务器验收记录。

使用规则：

1. 每位成员只把资料先放入本人角色目录的 `待检查/`。
2. 同步填写该角色目录下的 `资料登记表.csv`。
3. 审核状态统一使用 `可入库 / 仅参考 / 待核验 / 不采用`。
4. 只有移动到 `已检查可入库/` 的资料，后续才允许进入岗位画像、问答经验、能力模型、面试追问、学习路线、案例标注或答辩材料。
5. 大型文件、视频、压缩包不直接放入仓库；只记录链接、摘要和来源说明。
6. 原始资料和协作登记继续放在本目录；正式候选入库数据统一放入 `ai-interview-backend/app/data/knowledge_packages/`，避免把可部署数据长期存放在比赛资料目录。
7. 成员资料正式入库时使用后端脚本校验和导入，不直接手工写数据库；图片型资料必须 OCR 或人工转写后再进入候选包。

## 阶段 136：简历解析兼容性与开始面试稳定性升级

### 当前定位

本阶段解决“PDF 简历解析页面显示成功，但开始模拟面试闪退或报后端异常”的问题。判断重点不是盲目把所有简历都走 OCR，而是先建立稳定的 PDF 文本抽取、抽取质量诊断、结构归一和开始面试兜底。

### 已实施初版

- 新增统一简历文本抽取层：
  - 优先使用 `pdfplumber`。
  - 其次使用 `PyPDF2` 作为备用抽取结果。
  - 自动选择质量更高的文本结果。
  - 对空文本、极低字符数或结构缺失的 PDF 才建议进入 OCR fallback。
- 新增可选 OCR fallback：
  - 通过 `ENABLE_RESUME_OCR=true` 控制。
  - Dockerfile 已加入 `tesseract-ocr`、`tesseract-ocr-chi-sim`、`tesseract-ocr-eng`。
  - Python 依赖新增 `PyMuPDF`。
  - OCR 不作为普通文字版 PDF 默认路径，避免拖慢解析。
- 新增抽取质量诊断字段：
  - `analysis.extraction_diagnostics`
  - `analysis.parse_quality`
  - 记录页数、字符数、最佳抽取方式、是否 OCR、字段缺失、低质量原因和告警。
- 新增简历结构归一层：
  - `analysis.normalized_resume`
  - 稳定输出 `profile / education / projects / experience / skills / awards / campus_experience / raw_sections`。
  - 兼容 AI 返回的字符串、dict、list 混合结构。
  - AI 字段缺失时用规则从简历文本兜底抽取教育、项目、技能、联系方式、获奖和校园经历。
- 开始面试链路增强：
  - 旧简历如果没有 `normalized_resume`，开始面试时会现场归一。
  - `_resume_route_text` 会优先读取归一结构，再兼容旧字段。
  - 如果技能、项目或经历信息都不足，返回可读错误，不让前端闪退。
- 前端上传/分析页增强：
  - 显示解析质量、抽取方式、OCR 是否使用、待补字段和告警。
  - 开始面试失败时保留页面状态，并给出解析质量、OCR、数据库或模型配置方向的提示。

## 阶段 137：PostgreSQL 并发治理与数据库容量自检

### 背景

- 当前项目主数据库是 PostgreSQL，不是 `postersql`。
- 项目已经使用 `asyncpg`、Alembic、PostgreSQL `JSONB`、岗位知识切片、报告快照和学习任务 JSON 数据。
- 多人并发的近期风险主要来自连接池、Uvicorn worker、Celery、LLM 调用耗时和服务器资源，不宜直接用 MySQL 迁移绕过。

### 本轮处理

- 保留 PostgreSQL，不执行 MySQL 迁移。
- 后端数据库连接池从硬编码改为环境变量配置：
  - `DB_POOL_SIZE`
  - `DB_MAX_OVERFLOW`
  - `DB_POOL_TIMEOUT`
  - `DB_POOL_RECYCLE`
  - `DB_SCHEDULER_POOL_SIZE`
  - `DB_SCHEDULER_MAX_OVERFLOW`
- 主应用和调度任务分别使用独立连接池配置，避免小型服务器在多 worker 下放大数据库连接数。
- 新增只读脚本 `python -m app.scripts.check_database_capacity`，输出连接数、活跃连接、核心表行数、最大表、长运行查询和容量建议。
- `.env.example` 和后端 `.env.example` 增加 PostgreSQL 连接池推荐配置。

### 验证命令

```powershell
cd D:\apps\ai-interview\ai-interview-backend
python -m compileall app
python -m app.scripts.check_database_capacity
```

服务器执行：

```bash
cd /opt/apps/ai-interview
docker compose exec app alembic upgrade head
docker compose exec app python -m app.scripts.check_database_capacity
docker compose logs --tail=200 app | grep -Ei "too many connections|TimeoutError|asyncpg|Traceback"
```

### 当前状态

- 阶段 137 为 PostgreSQL 稳定性初版。
- 若连接数接近上限，先降低 `UVICORN_WORKERS`、`DB_POOL_SIZE` 或 `DB_MAX_OVERFLOW`。
- 若 LLM 调用慢，优先做任务排队、超时和重试。
- 若真实用户持续增加，再评估 PgBouncer、云数据库或独立 MySQL 迁移阶段。

## 阶段 139：授权转化、后台准入与学习任务质检小功能升级

### 当前定位

阶段 139 是阶段 138 前的产品小功能收口。目标不是替代真实闭环，而是提升后续数据沉淀质量：用户端更愿意授权，后台更清楚哪些案例可标注/可导出，学习任务在跑测前能暴露空泛问题。

### 本轮处理

- 用户端授权卡：
  - 优化报告页和训练复盘页复用的本次案例授权卡文案。
  - 明确授权后可针对性升级岗位画像、追问策略、评分规则、报告建议和学习任务质量。
  - 保持默认不勾选、自愿授权、可撤回、不影响核心功能。
- 后台案例标注：
  - 案例列表增加授权统计、授权筛选和授权状态列。
  - 未授权案例可查看详情，但不能保存人工标注。
  - 已授权案例显示可进入人工评分沉淀。
- 评测样本页：
  - 将样本口径改为“已授权完成测评”。
  - 固定规则展示 `data_contribution_consent=true`。
  - 空态提示改为需要完成面试、本次案例授权和后台人工标注。
- 学习任务页：
  - 增加任务质量状态：高质量、可用、待完善。
  - 质量依据固定为学习材料、练习任务、验收方式、预计耗时、来源依据或路线阶段。
  - 增加任务质量筛选，并在任务卡和学习计划草稿里提示缺失项。
