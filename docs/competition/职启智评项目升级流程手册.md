## 0. 总路线纠偏

当前项目不是从零做一个“人力资源岗位测评小工具”，而是在原 `ai-interview` 多岗位模拟面试系统上做参赛增强。

正确结构：

```text
原系统主线：用户设定岗位 -> 选择岗位画像/知识库 -> 开始模拟面试 -> 生成报告
    ↓
参赛增强：岗位画像库 + 轻量匹配评分 + 证据追问 + 人工评分对比 + 微调数据准备
```

因此后续所有阶段都必须遵守：

- 保留技术岗，例如 Python 后端、Java 后端、前端开发、测试工程师、算法工程师、数据分析师。
- 保留非技术岗，例如产品助理、人力资源专员、招聘助理、运营助理、行政管理助理、新媒体运营。
- 不把“大学生求职能力评估”覆盖成全站唯一语义，它是参赛增强功能、报告维度和展示页面。
- 技术深度不靠堆词，而要落到可运行指标、样本、对比和导出数据。

## 11. 阶段 8：Embedding 可选增强

当前状态：实施初版，待真实 API 验收。

### 操作步骤

当前采用“DeepSeek 优先、OpenAI 兜底、失败回退 TF-IDF”的方式接入：

```text
简历文本 -> embedding
岗位画像 -> embedding
cosine similarity -> embedding_semantic_score
```

要求：

- 有 DeepSeek API Key 且配置 `DEEPSEEK_EMBEDDING_MODEL` 时，优先尝试 DeepSeek OpenAI-compatible embeddings。
- 如果 DeepSeek embedding 不支持、模型未配置、请求失败或额度不足，则不中断简历分析。
- 如果用户同时保存 OpenAI Key，则尝试 OpenAI embedding 兜底。
- 无 API Key 或调用失败时回退 TF-IDF。
- 报告里明确展示使用的是 embedding 还是 TF-IDF fallback。

### 交付物

- `embedding_status` 字段。
- `tfidf_semantic_score`、`embedding_semantic_score`、`semantic_backend`、`embedding_provider`、`embedding_model`、`embedding_error_code` 字段。
- embedding 与 TF-IDF 对比记录。

### 验收标准

- API 不可用时系统不崩。
- `semantic_score` 继续作为前端主语义分：embedding 成功时取 embedding 分，否则取 TF-IDF 分。
- 简历分析页和报告页能显示“DeepSeek embedding / OpenAI embedding / TF-IDF 回退”。
- 答辩时能说明深度学习应用来自 LLM 与 embedding，不是自研底层模型。

### 当前验收记录

截至 2026-05-09，本阶段已完成初版代码落地：新增可选 embedding 增强服务；扩展 `matching_metrics` 字段；简历分析链路在写入 `analysis` JSON 前会尝试 embedding 增强；用户端简历分析页和报告页已显示语义来源。已通过 `python -m compileall app`、`pytest tests/test_embedding_match_service.py -q`、`pytest tests/test_embedding_match_service.py tests/test_ai_service_panel_structure.py tests/test_interview_service_round_flow.py -q`，其中结果为 24 个通过、1 个原有异步用例因缺少 `pytest-asyncio` 跳过；用户端和后台端临时构建均通过。本轮没有真实联网调用 DeepSeek embedding，也没有启动本地三端系统，因此仍需配置真实 `DEEPSEEK_EMBEDDING_MODEL` 和有效 API Key 后做阶段 1/13 浏览器验收。

### 不通过处理
