## 0. 总路线纠偏

当前项目不是从零做一个“人力资源岗位测评小工具”，而是在原 `ai-interview` 多岗位模拟面试系统上做参赛增强。

正确结构：

```text
原系统主线：用户设定岗位 -> 选择岗位画像/知识库 -> 开始模拟面试 -> 生成报告
    ↓
参赛增强：岗位画像库 + 轻量匹配评分 + 证据追问 + 人工评分对比 + 微调数据准备
```

因此后续所有阶段都必须遵守：

- 保留技术岗，例如 Python 后端、Java 后端、前端开发、测试工程师、算法工程师、数据分析师。
- 保留非技术岗，例如产品助理、人力资源专员、招聘助理、运营助理、行政管理助理、新媒体运营。
- 不把“大学生求职能力评估”覆盖成全站唯一语义，它是参赛增强功能、报告维度和展示页面。
- 技术深度不靠堆词，而要落到可运行指标、样本、对比和导出数据。

## 9. 阶段 6：Prompt 与证据约束

- 简历只写“参与项目”时，系统会追问个人职责、结果和指标。
- 简历没有 Redis 经历时，不会说候选人“熟练 Redis”。

### 不通过处理

如果模型编造，先降低 temperature，并在 Prompt 中强化“不编造”和“缺证据保守判断”。

## 10. 阶段 7：轻量机器学习匹配

### 操作步骤

当前先落地不依赖重库的指标：

- `keyword_coverage`：岗位关键词覆盖率。
- `semantic_score`：TF-IDF/余弦相似度。
- `rule_score`：规则评分。
- `final_score`：组合最终分。
- `evidence_basis`：可解释依据。

这些字段写入简历 `analysis` JSON。

### 交付物

- 简历分析页展示匹配指标。
- 后端导入检查能跑出这些字段。

### 验收标准

- 简历分析返回 JSON 中包含 `keyword_coverage`、`semantic_score`、`rule_score`、`final_score`、`evidence_basis`。
- 页面能展示这几个指标。
- 技术岗和非技术岗都能生成指标。

### 不通过处理

如果指标缺失，先检查 `resume_service` 是否把匹配引擎结果写入 `analysis`。

## 11. 阶段 8：Embedding 可选增强

### 操作步骤

后续在轻量评分稳定后接入：

```text
简历文本 -> embedding
岗位画像 -> embedding
cosine similarity -> embedding_semantic_score
```

要求：

- 有 API Key 时调用 embedding。
- 无 API Key 或调用失败时回退 TF-IDF。
- 报告里明确展示使用的是 embedding 还是 TF-IDF fallback。

### 交付物

- `embedding_status` 字段。
- embedding 与 TF-IDF 对比记录。

### 验收标准

- API 不可用时系统不崩。
- 答辩时能说明深度学习应用来自 LLM 与 embedding，不是自研底层模型。

### 不通过处理

短期来不及接 embedding 时，不说已经完成，只说当前完成 TF-IDF fallback 和规则评分。

## 12. 阶段 9：简历分析主流程升级

### 操作步骤

1. 上传 PDF 简历。
2. 输入目标岗位。
3. 等待解析完成。
4. 检查简历解析、证据摘要、匹配指标、改进建议。

### 交付物

- 一份技术岗简历分析结果。
- 一份非技术岗简历分析结果。

### 验收标准

- 技术岗案例能出现技术关键词匹配与缺口。
- 非技术岗案例能出现流程、沟通、数据、复盘等匹配与缺口。
- 页面展示不混淆岗位语境。

### 不通过处理

如果技术岗结果出现 HR 语境，检查默认岗位、岗位画像和 Prompt 是否误注入。

## 13. 阶段 10：模拟面试闭环升级

### 操作步骤

1. 选择目标岗位画像。
2. 选择 3-5 题。
3. 开始模拟面试。
4. 至少回答 3 轮。
5. 查看追问是否围绕岗位知识和简历证据。

### 交付物

- 一场技术岗面试记录。
- 一场非技术岗面试记录。
- 一份岗位面试评估报告。

### 验收标准

- Python 后端会问项目、数据库、缓存、接口、排障等问题。
- 产品/运营/HR 会问需求、活动、招聘、沟通、复盘等问题。
- 回答模糊时系统会追问证据，而不是泛泛鼓励。

### 不通过处理

问题跑题时，先检查岗位画像是否选择成功，再检查知识切片和 Prompt。

## 14. 阶段 11：后台样本与人工评分

### 操作步骤

后台进入面试详情：

- 标记是否适合作为样本。
- 填人工评分。
- 填审核结论。
- 导出评测样本。

### 交付物

- 至少 5 条样本。
- AI 评分与人工评分对比表。
- 后续 fine-tune-ready JSONL 数据。

### 验收标准

- 样本覆盖技术岗与非技术岗。
- 每条样本都有目标岗位、AI 分、人工分、差值、建议。
- 能说明人工评分依据。

### 不通过处理

如果后台数据不足，先用演示案例和手动记录表补足答辩材料，真实样本后续继续积累。

## 15. 阶段 12：测试案例与稳定性

### 操作步骤

至少准备 5 个案例，推荐扩展到 10 个：

- Python 后端
- 前端开发
- 测试工程师
- 数据分析师
- 产品助理
- 人力资源专员
- 运营助理

每个案例重复运行 3 次，记录：

- 关键词覆盖率
- 语义分
- 规则分
- LLM 评分
- 最终分
- 人工分
- 分数波动

### 交付物

- `AI评分与人工评分对比.xlsx` 或同结构表格。
- `测试案例与人工评分对比.md`。

### 验收标准

- 技术岗和非技术岗都有样例。
- 评分差值能解释。
- 波动过大时有调参记录。

### 不通过处理

分数不稳定时先固定模型、temperature、Prompt 版本，再重复测试。
