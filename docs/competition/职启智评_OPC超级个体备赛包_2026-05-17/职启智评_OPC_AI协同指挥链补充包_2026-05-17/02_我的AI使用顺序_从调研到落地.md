# 02 我的 AI 使用顺序：从调研到落地

## 1. 固定 8 步

```text
1. 我提出目标和约束
2. ChatGPT 分析赛道/项目/技术路线
3. ChatGPT 生成 Codex 可执行任务包
4. Codex 读取项目记忆和当前代码状态
5. Codex 生成或修改代码、文档、脚本、展示资产
6. Codex 运行测试/构建/静态检查并记录结果
7. 我再用 ChatGPT 把工程结果转成答辩话术
8. 我本人验收，决定提交、部署或继续补实
```

## 2. 第一步：我提出目标和约束

示例：

```text
我要参加京彩大创 OPC 超级个体赛道。
评委重点看 AI 工作流构建、人机协同逻辑设计、产业应用结构表达。
请基于 AI-Interview / 职启智评项目，重新设计参赛叙事，重点体现我作为一人 OPC 如何调度 ChatGPT、Codex、业务大模型和后台评分。
不要只讲产品功能，要讲 AI 之间如何协作。
```

重点：

```text
比赛目标
评委标准
项目背景
输出形式
不能夸大的边界
```

## 3. 第二步：ChatGPT 做策略分析

ChatGPT 应输出：

```text
赛道理解
项目适配判断
叙事重构
技术亮点
评委追问
Codex 任务拆解
```

Prompt 侧重点：

```text
不要泛泛夸项目。
把“AI 面试平台”重构为“OPC AI 工作流”。
把分析结果转成 Codex 可以执行的任务。
```

## 4. 第三步：ChatGPT 生成 Codex 任务包

任务包必须包含：

```text
读哪些文件
新增哪些目录
生成哪些 Markdown
是否新增前端只读页
是否新增后端只读接口
怎么更新任务记忆
怎么验证
不能做什么
```

## 5. 第四步：Codex 读取项目记忆

Codex 每轮必须先读：

```text
PROJECT_MEMORY.md
任务记录文档.md
docs/competition/职启智评项目升级流程手册.md
```

原因：

```text
防止重复造轮子
防止破坏旧功能
防止忽略部署和目录约束
保证每轮工作可追溯
```

## 6. 第五步：Codex 执行工程任务

Codex 做：

```text
新增 Markdown
新增脚本
生成 demo case
生成 agent trace
生成 eval preview
生成 sft preview
新增只读展示页
修改路由
补测试
补任务记忆
输出 commit 建议
```

## 7. 第六步：Codex 验证

可能执行：

```text
python -m py_compile
pytest
npm run build
docker compose config
alembic upgrade head
curl health
脚本 dry-run
rg 静态检查
```

跑不了也要写：

```text
未完成验证：原因是什么，下一步如何补。
```

## 8. 第七步：ChatGPT 转译答辩语言

示例：

```text
Codex 输出：新增 position_knowledge_base_slice_service.py。
答辩表达：系统将岗位资料自动切片并按面试阶段召回，避免整份知识库直接塞进 prompt，提高追问相关性和可解释性。
```

## 9. 第八步：我最终验收

我判断：

```text
是否贴合比赛
是否能现场演示
是否破坏原项目
是否有夸大风险
是否写入 PPT
是否需要继续交给 Codex
```
