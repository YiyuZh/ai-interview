## 二、项目概述

职启智评 OPC 面向高校学生求职准备中“岗位认知不清、简历证据不足、面试训练零散、反馈难复盘”的问题，构建一个由个人负责人调度多类 AI Agent 的就业能力诊断与训练服务单元。

项目不是单次调用大模型生成面试题，也不是普通 AI 面试官包装，而是把高校就业指导中的岗位理解、简历诊断、证据评分、简历润色、面试追问、报告复盘、学习任务和数据治理拆解为可编排、可追踪、可复盘的 AI 原生工作流。

在 OPC 口径下，负责人作为 `OPC Commander`，负责目标选择、真实性判断、隐私授权、评分标准和最终验收；ChatGPT、Codex、业务大模型、Eval/后台评分分别承担赛道研究、工程实现、业务生成、质量评估和材料沉淀。

## 三、AI 原生结构总览

### 3.1 总体结构

```text
OPC Commander（本人）
  -> ChatGPT：赛道研究、策略拆解、Prompt 与答辩材料
  -> Codex：代码、脚本、测试、部署、文档和 Git 记录
  -> 业务大模型 Agent：简历解析、岗位画像、润色、追问、报告和学习任务
  -> Eval / 后台评分：规则评估、人工复核、样本准入和风险发现
  -> OPC Commander：真实性、授权、边界和最终提交
```

### 3.2 AI 工作流闭环

| 环节 | 输入 | AI / 系统处理 | 输出 | 人工验收点 |
|---|---|---|---|---|
| 赛道与任务拆解 | 赛事通知、项目现状 | ChatGPT 提炼 OPC 评审重点和任务包 | OPC 叙事、PPT Brief、问答库 | 是否符合项目真实状态 |
| 工程落地 | Codex 任务包、项目代码 | Codex 修改代码、生成脚本、跑测试 | Git 提交、脚本、页面、文档 | 是否破坏主链路 |
| 简历诊断 | 简历、目标岗位、岗位画像 | 业务大模型抽取结构化信息并对齐岗位要求 | 证据评分、能力差距、快照 | 是否把岗位要求误写成候选人经历 |
| 简历润色 | 证据状态、岗位画像、能力差距 | 简历润色 Agent 生成“可改但不造假”的表达建议 | 润色建议、补证据建议 | 是否新增不存在的公司、项目、指标 |
| 面试训练 | 能力缺口、候选人回答 | 面试追问 Agent 围绕证据不足处追问 | 多轮问答、报告、复盘 | 是否围绕真实证据和目标能力 |
| 数据治理 | 授权、人工评分、PII 扫描 | Eval / 后台评分做准入和风险检查 | 可沉淀样本、拒绝原因 | 是否满足授权、复核、脱敏规则 |

## 四、人机协同机制

本项目的创新点不是让 AI 完全替代人，而是把原本需要产品、工程、就业顾问、测试、运营和材料团队协作完成的工作，拆解成一个人可以调度的 AI 协同结构。

| 事项 | AI 负责 | 人负责 |
|---|---|---|
| 目标判断 | 根据资料提出方案和风险点 | 决定项目方向、优先级和最终口径 |
| 内容生成 | 生成岗位画像、追问、报告、学习任务和材料草稿 | 审查真实性、可执行性和合规性 |
| 工程执行 | 修改代码、运行测试、整理文档 | 选择是否合并、提交、部署 |
| 质量评估 | 规则评分、PII 提示、样本准入建议 | 制定评分标准并做最终确认 |
| 数据使用 | 处理已授权、去标识化后的数据 | 控制授权、撤回、人工复核和边界说明 |

该机制保证 AI 承担高频生成、检索、工程执行和初步评估，人保留目标、授权、真实性、评分标准和最终责任。

## 五、Career-AgentOS 工作流

职启智评的业务底座已经从单点 AI 面试升级为 Career-AgentOS 就业能力诊断闭环。当前主链路如下：

```text
简历解析 -> 岗位画像 -> 证据评分 -> 能力差距 -> 简历润色 -> 面试追问 -> 报告复盘 -> 学习任务 -> 数据治理 -> Eval Preview -> SFT-ready 门禁
```

### 5.1 证据约束

系统把候选人信息区分为 `direct`、`indirect`、`claimed_only`、`missing` 等证据状态。岗位知识库只能作为岗位要求、追问方向和表达参考，不能写成候选人的真实经历。

### 5.2 简历润色 Agent

简历润色不是编造经历，而是在已有证据基础上进行岗位化表达。缺少证据时，系统只能给出补证据建议或占位表达，不能新增公司、项目、时间、证书、百分比、技术栈或指标。

### 5.3 证据追问型面试

面试追问围绕能力缺口和证据不足展开，重点追问候选人在具体场景中的任务、行动、结果、指标和复盘，而不是随机生成通用问题。

### 5.4 数据治理与样本准入

样本进入评测或训练前需要满足：用户本次案例授权、面试完成、后台人工复核、PII 扫描通过、样本来源明确。不同意数据贡献不影响核心功能，但不能进入评测或训练样本沉淀。

## 七、复赛集中限时实测能力

OPC 复赛可能要求参赛者在限定时间内完成新问题拆解、人机协同重构和垂直场景落地。职启智评当前准备了三类训练：

| 题型 | 限时 | 输出物 |
|---|---:|---|
| 构建新岗位 AI 工作流 | 30 分钟 | 新岗位画像、Agent 分工、质量门禁 |
| 重构人机协同结构 | 45 分钟 | 人负责什么、AI 负责什么、如何验收 |
| 垂直场景迁移方案 | 60 分钟 | 从高校就业服务迁移到相邻训练场景 |

标准作答框架为：

```text
识别场景和用户 -> 拆解业务流程 -> 标注人机边界 -> 分配 AI Agent -> 定义输出物 -> 设置质量和合规门禁 -> 给出三个月补实路线
```

## 八、产业应用场景

项目面向高校就业服务、大学生求职训练、学院就业辅导、创新实践课程和个人求职准备。

现有就业指导高度依赖老师经验和线下咨询，覆盖频次有限；普通 AI 简历润色或 AI 面试产品多为单点功能，缺少岗位画像、证据约束、学习闭环和人工校准。职启智评通过 AI 原生工作流，把一次求职准备拆成可重复执行、可审查、可复盘的训练过程。

## 九、主体治理与合规边界

本项目按个人主体申报。负责人承担项目方向、工程实现、系统运行、材料真实性和最终答辩责任。其他协作材料只作为资料支持，不夸大为完整商业团队能力。

合规边界如下：

- 使用基础隐私协议和可选数据贡献授权区分核心服务与样本沉淀。
- 使用去标识化 / 脱敏表述，不宣称完全匿名化。
- 不使用未授权数据进入训练或评测样本。
- 不把岗位知识库内容写成候选人真实经历。
- 不把构造 demo 样本写成真实用户样本。
