## 01 任务目标

本任务要解决的问题：

- 将 `AI-Interview` 从原本的本地 / VM 开发式部署，改造成可在腾讯云 Linux 服务器上稳定运行的生产部署方案。
- 保留原有项目组织方式与部署思路，不推翻重来：
  - 保留 `backend / frontend / admin` 三分结构
  - 保留 `.env`
  - 保留 `Docker Compose`
  - 保留 `Alembic`
- 让 `AI-Interview` 与现有 `HireMate` 在同一台腾讯云服务器上共存，并优先保证 `HireMate` 不受影响。
- 使用现有 `hiremate-caddy` / 统一 `Caddy` 作为唯一公网入口。
- 增加主域名项目选择页 `portal`。
- 把前端从“本地 Vite dev server”改造成“服务器构建 + Docker 静态部署”模式，同时保留开发环境运行方式。
- 修复上线后遇到的注册失败、登录失败、接口 405、容器 unhealthy、配置填写不明确等问题。
- 优化注册 / 登录速度，优先保证稳定性。
- 在不推翻现有项目结构的前提下，为每个用户新增“岗位知识库”能力，并把知识库真正接入 AI 面试官的提问、追问和报告生成链路。
- 在已有用户岗位知识库的基础上，再增加“后台公共知识库”能力，让普通用户可以使用后台发布的知识库，但不能编辑它们。

### 当前项目技术栈

- 后端：FastAPI（异步）
- 前端：Vue 3 + Vite
- 管理端：Vue 3 + Vite
- AI：DeepSeek API（兼容 OpenAI SDK）
- 数据库：PostgreSQL 16
- 缓存 / 队列：Redis 7
- 异步任务：Celery Worker + Celery Beat
- ORM：SQLAlchemy 2.0 + asyncpg
- 部署：Docker + Docker Compose
- 统一入口：Caddy
- 前端静态容器：Nginx

### 当前产品定位

- `AI-Interview / 智面` 的核心定位不是单纯聊天机器人，而是“基于简历和目标岗位的 AI 模拟面试训练平台”。
- 用户侧主业务闭环目前是：
  1. 注册 / 登录
  2. 上传 PDF 简历
  3. AI 解析简历并生成结构化信息
  4. AI 分析简历与目标岗位的匹配度
  5. 用户开始模拟面试
  6. AI 面试官逐题提问并即时点评
  7. 最终生成综合评估报告
- 管理端的定位是后台运营与数据查看，不是主要业务创新位。
- 因此，本项目后续最值得投入的功能增强点，应优先围绕“让提问更像真实岗位、让追问更有训练价值、让报告更可执行”展开。

## 03 已做决策 + 理由

做了什么选择，为什么：

### 决策 1：不推翻原部署路线，只做生产化改造

理由：

- 用户明确要求保留原部署文档的整体思路
- 可最大限度减少认知成本与迁移风险

### 决策 2：继续保留 `.env + Docker Compose + Alembic`

理由：

- 这是用户原文档的核心工作流
- 便于后续日常维护、上线、回滚

### 决策 3：继续保留 `backend / frontend / admin` 三分结构

理由：

- 这已经是现有项目的事实结构
- 管理端与用户端职责清晰，不应合并

### 决策 4：统一复用现有 `Caddy` 作为唯一公网入口

理由：

- `HireMate` 已在使用 `Caddy`
- 避免重复占用 `80/443`
- 更利于多项目共存与后续扩展

### 决策 5：所有站点继续通过 `shared_gateway` 互联

理由：

- 与现有多项目部署路线一致
- 便于 `Caddy` 反代多个容器

### 决策 6：数据库 / Redis / FastAPI 不对公网暴露

理由：

- 生产环境安全要求
- 降低误暴露与被扫风险
- 统一通过 `Caddy` 和容器内网访问

### 决策 7：主域名先展示 `portal`，再跳转具体项目

理由：

- 用户明确要求访问主域名先看到项目选择页
- 适合一台服务器承载多个产品

### 决策 8：前端生产部署不再依赖 `vite.config.js` 写死 VM IP

理由：

- 原方式只适合 VM / 本地联调
- 生产环境必须走环境变量注入和反向代理

### 决策 9：当前首发注册模式默认使用“直接注册即登录”

理由：

- 更适合首发阶段
- 不依赖 SMTP / 邮件验证码链路
- 能先跑通主业务闭环

对应开关：

- `AUTH_REQUIRE_EMAIL_VERIFICATION=false`

### 决策 10：邮箱验证码注册作为未来开关，不默认启用

理由：

- 用户希望未来可接入，但当前先稳定上线
- 减少 Redis / Celery / SMTP 故障面

### 决策 11：预览阶段前端 nginx 也必须反代 `/api`

理由：

- 否则用户通过 SSH 隧道访问 `13000/13001` 时，请求会落到静态 Nginx，返回 405
- 这已经在真实排查中被验证

### 决策 12：注册 / 登录慢主要按代码路径优化，而不是先归因于服务器

理由：

- 实际检查后发现认证链路存在明显代码级重开销：
  - refresh token 也走 `bcrypt`
  - 密码哈希 / 校验阻塞 async 线程
  - 生产仍然打印 SQL echo
- 服务器弱会放大问题，但不是首要根因

### 决策 13：refresh token 改用 HMAC-SHA256，兼容旧 bcrypt token

理由：

- refresh token 本身已经是高熵随机 JWT
- 使用 `bcrypt` 对 refresh token 哈希成本过高
- 改用 `HMAC-SHA256` 后可以显著减少登录 / 注册延迟
- 保留旧 token 校验兼容，避免线上用户立刻失效

### 决策 14：密码哈希 / 校验改为 `asyncio.to_thread(...)`

理由：

- `bcrypt` 是 CPU 密集型
- 直接在 async 请求线程中运行会阻塞事件循环
- 移到线程更稳定

### 决策 15：生产环境关闭 SQLAlchemy `echo=True`

理由：

- 注册 / 登录是高频接口
- 生产打印 SQL 明显增加日志 IO 和性能负担

### 决策 16：岗位知识库先采用“结构化文本知识库”，不直接引入向量库 / RAG

理由：

- 当前项目的最强主链路是“简历 + 岗位 + 面试 prompt”，不是文档检索系统
- 直接上向量库会显著增加部署复杂度、召回调优成本和线上不确定性
- 对现阶段的面试训练场景而言，先用用户可编辑的结构化文本知识库注入 prompt，已经能显著提升提问质量

### 决策 17：每个用户可维护多份岗位知识库，但每场面试只绑定一份知识库快照

理由：

- 多份知识库能覆盖不同岗位或同岗位不同专题
- 单场面试只绑定一份快照，能避免中途修改知识库导致同一场面试前后标准漂移
- 这比“实时始终读取最新知识库”更稳定、更可回溯

### 决策 18：岗位知识库放在用户端独立页面管理，并在开始面试前可选注入

理由：

- 不破坏现有“上传简历 -> 开始面试”的主流程
- 用户仍然可以不使用知识库，保持原有体验
- 当用户已经整理过岗位知识库时，可以在开始面试前显式选择，心智最清晰

### 决策 19：面试官 prompt 升级为“简历 + 目标岗位 + 用户知识库 + 面试风格要求”的联合提示

理由：

- 原本 prompt 只依赖简历和岗位，提问会偏通用
- 加入用户自定义岗位知识库后，提问能更贴近真实考点
- 加入“额外面试官要求”后，用户可以控制面试风格，例如更严格、更爱追问原理、更注重项目落地

### 决策 20：保留用户自建知识库，不因为数据库顾虑而删掉

理由：

- 当前知识库存储的是少量结构化文本，并不是大规模附件或向量数据
- 从 PostgreSQL 角度看，这类记录量对数据库压力很轻，不构成当前阶段的主要风险
- 用户自建知识库对训练价值很高，能显著提升个性化程度，不应过早砍掉

### 决策 21：新增后台公共知识库，形成“私有 + 公共”双层模型

理由：

- 用户自建知识库适合个性化训练，但内容质量参差不齐
- 后台公共知识库适合沉淀高质量标准题库和岗位模板
- 两层并存，既保留个性化，也能保证平台侧可控的基础训练质量

### 决策 22：普通用户只能编辑自己的私有知识库，但可以使用后台公共知识库

理由：

- 这能满足权限隔离：用户之间互不影响
- 同时让平台运营侧可以统一投放高质量知识库给所有用户使用
- 这比“只保留后台知识库”更灵活，也比“所有知识库都开放编辑”更安全

   - 原因：当时仍是本机预览阶段，只监听 `127.0.0.1:18080`

   - 处理：手册改为“照填版”

   - 处理：说明这是 Compose 服务名，不是 shell 命令

   - 处理：
     - 后端配置增加空字符串兜底回退
     - `.env.example` 和手册改为安全默认值

6. `pg_dump` 报 `option requires an argument: d`
   - 原因：文档中的命令换行导致 `-d` 与库名被拆开
   - 处理：手册改为整行可直接复制

7. 用户不清楚 `HireMate` 迁移时旧 MySQL 密码怎么确定
   - 处理：从旧 compose 默认值、旧卷名、inspect 路径中推断，并明确先兼容旧值

8. SSH 隧道命令在服务器里执行、且尖括号占位符未替换
   - 处理：说明必须在本地终端执行，并替换为真实 IP / 用户

9. 首次 SSH 停在 `password:`，用户误以为报错
   - 处理：说明这是正常密码登录提示
