# AI-Interview / 智面短期升级方案

更新时间：2026-04-18

## 1. 当前判断

AI-Interview 当前不应再把短期重点放在“换模型”或“立即微调”上。结合任务记录文档，最近已经连续完成了三层证据化能力：

- Resume Evidence Layer：简历解析后生成 `resume_evidence / evidence_summary`。
- JD Gap Graph + Interview Blueprint：开始面试前生成 `interview_blueprint`，把简历证据、目标岗位、知识库切片合成面试策略。
- Evidence-driven Follow-up Loop：每轮回答后记录证据强度变化、薄弱缺口、下一步最佳追问，并汇总进报告。

因此短期主线要从“继续堆提问能力”升级为：

**证据化面试闭环稳定化 + 训练样本数据飞轮。**

微调仍然是中期方向，但必须建立在高质量样本、失败案例、人工可审查数据结构之上，不能直接把线上主链路切到微调模型。

## 2. 已完成阶段

### P0：稳定性与上线一致性

已完成重点：

- 保留现有 `backend / frontend / admin` 三分结构。
- 保留 `.env / Docker Compose / Alembic`。
- 继续复用 `hiremate-caddy / shared_gateway`。
- 注册、登录、简历上传、开始面试、报告等线上链路已多轮修复。
- 生产构建已明确使用国内 PyPI 镜像。

后续要求：

- 每次新增字段优先复用 JSON 容器；确需表结构变化时再补 Alembic。
- 所有新能力必须有 fallback，不允许阻断主链路。

### P1：RAG 证据化底座

已完成重点：

- 知识库切片具备 routing score / routing reasons。
- 题目级 `selected_slices / used_slice_ids / evidence_trace / evidence_summary` 已接入。
- 报告页已能展示训练依据。

### P2：无证据不强答

已完成重点：

- `question / followup / evaluation / report` prompt 已补 grounding 约束。
- 证据不足时要求保守输出，不允许编造候选人经历、项目细节、业务事实。
- 简历解析 JSON 容错已增强，失败时不直接卡死主链路。

### P3：多角色共享同一证据池

已完成重点：

- panel 模式保留“内部多角色 + 外部单主持人”。
- 多角色评估共享同一批 evidence。
- moderator 的选择与追问开始显式绑定 evidence source。
- panel 失败时可回退到 single evaluator。

### P4：证据化报告与追问闭环

已完成重点：

- 开始面试前生成 `interview_blueprint`。
- 每轮评估输出：
  - `evidence_strength_delta`
  - `claim_confidence_change`
  - `unresolved_gaps`
  - `next_best_followup`
- 最终报告聚合：
  - `common_gaps`
  - `common_strengths`
  - `training_priorities`
  - `followup_loop_summary`
  - `claim_confidence_summary`

## 3. 当前最优先：P5 数据飞轮

从现在开始，短期升级进入 P5：数据飞轮准备。

目标不是马上训练模型，而是先把每场面试沉淀成可审查、可导出、可标注的训练样本。

### P5-1：训练样本导出

目标：

- 从已完成面试中导出结构化 training sample。
- 样本必须包含题目、回答、评分、反馈、证据引用、追问闭环、报告聚合信号。
- 默认不暴露用户隐私信息。

最小实现：

- 后台接口增加单场面试训练样本导出。
- 后台接口增加批量导出最近完成面试样本。
- 不新增表，不加 migration。
- 样本结构版本化，便于后续迭代。

### P5-2：人工标注入口

目标：

- 后台能对样本标注质量等级。
- 先不做复杂平台，最小可用字段包括：
  - 是否高质量样本
  - 是否存在幻觉
- 是否追问有训练价值
- 是否报告可执行

已完成最小实现：

- 后台面试详情页新增“训练样本标注”区域。
- 标注结果复用 `panel_snapshot.training_sample_review` 存储，不新增表。
- 标注结果会自动进入训练样本导出结果，便于后续筛样本。
- 后台面试列表增加“已标注 / 待标注”状态。

### P5-3：离线评测集

目标：

- 从真实样本中沉淀黄金样本集。
- 覆盖不同岗位、不同简历质量、不同知识库命中情况。

已完成最小实现：

- `golden_cases.jsonl`
- `hallucination_cases.jsonl`
- `followup_quality_cases.jsonl`
- `report_quality_cases.jsonl`
- 后台新增独立“离线评测集”页面。
- 后端新增 preview / export 两个接口。
- 导出格式固定为 `ZIP + 多个 JSONL + manifest.json`。
- 第一版规则固定为“人工标注 + 分数门槛”，并允许样本重叠进入多个集合。

### P5-4：微调试点准备

只有当 P5-1 到 P5-3 跑通后，才进入微调试点。

微调范围优先选择低风险任务：

- 追问生成格式稳定化。
- 结构化评分 JSON 稳定化。
- 报告总结表达一致性。

不建议短期直接微调整个面试主模型。

## 4. 下一步执行顺序

当前立即执行：

1. 用真实已标注面试验证 P5-3 四类集合命中是否符合预期。
2. 开始进入 P5-4：低风险任务型微调试点准备。
3. 优先围绕追问生成稳定性、结构化评分稳定性、报告总结一致性做离线对比。

验收标准：

- 后台能导出单场面试 training sample。
- 能批量导出最近已完成面试样本。
- 样本中能看到：
  - `interview_blueprint`
  - `question_target_gap`
  - `question_target_evidence`
  - `evidence_strength_delta`
  - `claim_confidence_change`
  - `next_best_followup`
  - `common_gaps / common_strengths / training_priorities`
- 旧面试数据缺字段时也能导出，不报错。
- 后台可对单场面试保存人工标注，并在再次打开时回显。
- 导出的样本默认会带出 `training_sample_review`。
- 后台可预览四类评测集的样本数、示例面试 ID、命中规则。
- 后台可一键导出固定结构的评测集 ZIP。

## 5. 仍然不建议短期投入的事项

- 不直接上全链路微调。
- 不大改前端交互。
- 不新增重型向量数据库或复杂 RAG 基建。
- 不把 panel 做成多 AI 聊天 UI。
- 不重建数据库模型。

短期核心原则仍然是：

**最小侵入、稳定可回滚、证据优先、数据沉淀优先。**
