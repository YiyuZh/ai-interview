基于你这份任务记录文档，我判断这两个方向非常对路：当前项目已经把核心价值放在“提问更像真实岗位、追问更有训练价值、报告更可执行”，并且已经引入了用户私有知识库和后台公共知识库；所以再往前走，最合理的就是把“单面试官”升级成“多视角协同”，把“整份知识库注入”升级成“切片化按需调动”。

我建议不要把它做成“前台出现多个 AI 同时说话”，而是做成：

一个主持面试官 + 一个内部多面试官讨论层

这样最稳，也最符合你现在的项目结构。

1. 多面试官，最适合你当前项目的形态

建议先固定 4 到 5 个角色，不要一上来做得太花：

技术深挖官
盯原理、边界、性能、故障排查。
项目追问官
盯方案取舍、落地细节、结果复盘。
业务场景官
盯岗位真实场景、业务理解、决策思路。
行为表达官
盯沟通结构、协作、冲突处理、表达质量。
压力质询官
盯追问强度、抗压、临场反应。

但对用户可见的，始终只保留一个主持面试官。
也就是说，用户看到的是一条连续面试流程；内部其实是多个角色先各自产生判断，再由主持人汇总后发问、点评、写报告。

这比“多个聊天气泡同时发言”更适合你现在的前端，也不会把用户体验搞乱。

2. 多面试官的工作流，建议这样落

不是每个角色都直接发题，而是内部先协同：

出题前

输入：

简历结构化结果
目标岗位
当前面试阶段
当前轮相关知识库切片
候选人前面答题表现

内部先做一轮结构化协同：

技术深挖官：给出“该问什么、想验证什么”
项目追问官：给出“该往哪里继续追”
业务场景官：给出“真实岗位会怎么问”
行为表达官：给出“表达和沟通能怎么考”
压力质询官：给出“哪里适合做反问或施压”

然后主持面试官输出：

本轮主问题
1 到 2 个备选追问
本轮评分重点
本轮需要调动的知识片段 ID
用户回答后

内部再跑一轮评估：

技术深挖官：判断是否答到原理和边界
项目追问官：判断是否讲清取舍和结果
业务场景官：判断是否贴近真实业务
行为表达官：判断是否结构清晰、有说服力
压力质询官：判断是否扛得住追问

最后主持人输出给用户的内容只保留三层：

一句综合判断
两到三条关键问题
下一轮最值得改进的一点

这样训练效率最高，不会给用户塞一堆碎反馈。

3. 技术实现上，不建议一开始就“每轮多次并发调用模型”

第一阶段更稳的做法是：

一次模型调用里模拟多个角色，并要求输出结构化 JSON

比如让 ai_service.py 先产出这种中间结果：

{
  "panel": [
    {
      "role": "technical",
      "focus": ["缓存一致性", "高并发边界"],
      "question_candidates": ["你们当时为什么选 Redis 而不是本地缓存？"],
      "followup_candidates": ["如果 Redis 挂了你怎么降级？"],
      "evaluation_points": ["原理是否清楚", "异常场景是否考虑"]
    },
    {
      "role": "project",
      "focus": ["方案取舍", "落地结果"],
      "question_candidates": ["你在这个项目里最关键的技术决策是什么？"],
      "followup_candidates": ["有没有踩坑，后来怎么改？"],
      "evaluation_points": ["是否讲清背景-动作-结果"]
    }
  ],
  "moderator": {
    "selected_question": "你们当时为什么选 Redis，而不是本地缓存或数据库直查？",
    "selected_followups": [
      "如果 Redis 挂了，你怎么保证服务不雪崩？"
    ],
    "feedback_style": "严格、真实面试风格"
  },
  "retrieved_slice_ids": ["slice_12", "slice_37", "slice_51"]
}

然后再由系统把它整理成用户看到的话术。

这样做有几个好处：

不推翻现有 ai_service.py / interview_service.py
成本和时延比“多模型并发”低很多
日后真要升级成多调用，也能平滑演进
便于把“面试官内部讨论”存成结构化记录，用于最终报告
4. 知识库切片，建议从“文本块”升级成“语义切片”

你现在的知识库已经有了，但如果还是把整份知识库整体注入 prompt，很快会遇到两个问题：

命中率不稳定
token 浪费很大

所以这里不要简单按 500 字、1000 字切段，而是按“训练用途”切。

建议新增一张切片表，例如：

position_knowledge_base_slices

核心字段建议有：

id
knowledge_base_id
title
content
slice_type
topic_tags
skill_tags
scene_tags
question_stage_tags
role_tags
difficulty
priority
keywords
source_scope（private / public）
is_enabled

其中最关键的是这几个维度：

slice_type

不是普通 chunk，而是知识类型：

concept：核心概念
scenario：业务场景
pitfall：常见误区
followup_point：适合追问的点
rubric：评价标准
faq：高频问法
question_stage_tags

指定它适合哪个阶段：

question
followup
evaluation
report
role_tags

指定它更适合哪个面试官调用：

technical
project
business
behavioral
pressure

这一步一做，知识库就不是“长文本附件”了，而是“可路由的训练组件”。

5. 知识库切片的调动逻辑，建议做成混合召回

第一阶段不必上独立向量库，和你当前文档路线一致，先走轻量版就够了。

建议召回分数由这几项构成：

岗位匹配
当前问题阶段匹配
面试官角色匹配
技能标签匹配
简历弱点匹配
最近答错点匹配
关键词命中
手工优先级

也就是：

不是“搜索最像的文本”，而是“搜索当前这轮最该被调出来的知识片段”

比如：

出题阶段：优先拉 concept / scenario
追问阶段：优先拉 followup_point / pitfall
评分阶段：优先拉 rubric / pitfall
报告阶段：优先拉 rubric / scenario / common_gap

这样知识库就真正进入面试链路了，不只是“附带参考材料”。

8. 最好的开发顺序

我建议顺序不要反：

第一步

先做知识库切片表 + 切片召回服务

原因很简单：
多面试官没有“精准知识调动”，最后还是泛化提问。

第二步

在 ai_service.py 里加入多角色结构化协同输出

先不要多调用，先单调用模拟多角色。

第三步

把多角色结果接到：

逐轮问题
逐轮点评
最终报告
第四步

再补前端模式选择、切片预览、报告分视角展示

这个顺序最稳。
