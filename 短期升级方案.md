# AI-Interview 短期升级方案

## 1. 文档目的

这份方案不是推翻现有项目，而是基于当前 `AI-Interview / 智面` 已经落地的能力，给出一条更稳、更像产品升级而不是技术试验的短期路线。

当前目标不是“把模型变得更会聊天”，而是把系统升级成：

- 更少幻觉
- 更少答非所问
- 更贴近目标岗位
- 更像真实面试训练
- 更能稳定复用用户知识库与公共知识库

结论先写在前面：

**下一阶段最应该继续投入的方向是 RAG 证据化，而不是先重押模型微调。**

微调要做，但不应该是第一刀。

原因很简单：

- 当前线上问题更多来自“知识没有被稳定调出来”，而不是“模型完全不会”
- 当前项目已经有岗位知识库、切片、路由、多视角协同雏形，继续把这条链打实，收益最大
- 直接上微调，会很快遇到样本不足、标注不稳、上线回滚难、效果难归因的问题

所以短期路线建议定为：

**RAG 强化优先，微调准备同步推进，微调正式落地放在证据化链路稳定之后。**

---

## 2. 当前项目实际现状

结合当前仓库和已落地改动，项目已经具备这些基础：

### 2.1 产品定位已清晰

当前产品不是通用聊天机器人，而是：

**基于简历、岗位和知识库的 AI 模拟面试训练平台**

核心价值已经不是“能不能回答”，而是：

- 问得像不像真实岗位
- 追问有没有训练价值
- 点评是否有依据
- 报告能不能指导下一轮训练

### 2.2 已有能力基础

当前仓库里已经落地的关键基础包括：

- 用户端 / 管理端 / 后端三分结构已稳定
- 用户私有知识库与后台公共知识库都已接入
- 知识库切片能力已经开始落地
- 多角色内部协同 + 单主持人对外输出的一阶段结构已经存在
- 提问、逐轮评估、报告链路都已开始接 panel / slice 信息
- 前端已经做了最小增强：
  - 多面试官模式开关
  - 面试过程轻量摘要
  - 报告页多视角摘要
  - 知识库切片调试预览

### 2.3 当前最明显的产品短板

虽然基础不错，但从产品体验看，当前仍有几个明显问题：

1. 面试官仍可能出现幻觉
2. 提问和追问有时不够精确，容易泛化
3. 点评有时“像在评价”，但依据不够明确
4. 报告虽然比纯聊天强，但仍不够“证据驱动”
5. 知识库虽然已切片，但还没完全变成“可信知识调用系统”

这些问题的根本不是一个点，而是三个层面叠加：

- 检索命中率不够稳定
- prompt 虽有结构，但缺少“必须基于证据回答”的硬约束
- 当前还没形成高质量训练样本闭环，微调条件还不成熟

## 3. 产品判断：下一阶段的主方向

### 3.1 总体方向

下一阶段应该把 `AI-Interview` 从：

**“有知识库增强的 AI 面试官”**

升级成：

**“证据驱动的岗位训练面试系统”**

这意味着模型输出不再主要依赖“它自己会不会”，而是优先依赖：

- 当前岗位
- 当前知识库切片
- 当前轮次上下文
- 当前候选人薄弱点
- 当前多角色面试官内部判断

### 3.2 为什么 RAG 要放在微调前面

你提出的“继续做 RAG / 微调来修复幻觉和回答不精确”方向是对的，但优先级必须分清：

#### 先做 RAG 的原因

- 幻觉首先是 grounding 问题，不是参数问题
- 岗位面试本来就是强上下文场景，非常适合做检索增强
- RAG 的收益可解释、可调试、可回滚
- 当前项目已有知识库、切片、Top-K 路由，继续增强成本最低

#### 微调暂不作为第一优先的原因

- 当前项目还缺足够稳定的高质量标注数据
- 如果先微调，容易把错误风格一起学进去
- 微调对“岗位知识更新”并不灵活
- 线上出现问题时，微调效果比 RAG 更难定位

所以短期路线建议明确为：

**第一优先级：把 RAG 做成“证据化调用系统”**

**第二优先级：围绕 RAG 产出训练数据闭环**

**第三优先级：再做小范围、任务型微调**

## 4. 短期升级总路线

### 主线 A：证据化 RAG

目标：

- 让问题、追问、点评、报告都真正基于已召回证据
- 降低幻觉和泛化输出
- 让系统具备可解释性和可调试性

### 主线 B：多角色面试官继续保留，但必须建立在证据之上

目标：

- 让多个角色围绕同一批证据做不同视角判断
- 保持对外仍然只有单主持人输出
- 让多角色成为“训练价值放大器”，而不是复杂度放大器

### 主线 C：评估与报告证据化

目标：

- 报告不再只是总结，而是能说明“依据是什么”
- 高频短板、训练建议、追问方向都能追溯到轮次与证据

### 主线 D：微调准备，但不急于直接上生产主链

目标：

- 先积累高质量数据飞轮
- 先做离线评测
- 再决定是否微调、微调什么

## 4.1 执行任务清单（按优先级排序）

下面这份清单用于把本方案直接转成可执行升级任务，按“稳定性优先、证据化优先、最小侵入优先”的原则排序。

### P0：稳定性与一致性收口

1. 修复并验证生产数据库 migration 一致性
2. 确认 `schema / model / migration / 线上库` 完全一致
3. 收口 fallback 行为和关键日志，确保切片失败、panel 失败、报告失败时都有稳定降级路径

### P1：证据化 RAG 底座

4. 给切片召回结果补“命中原因 / 命中摘要 / routing score”等可解释信息
5. 把每道题的 `selected_slices / used_slice_ids` 升级成可追踪的 `evidence_trace`
6. 把证据摘要接入最终报告，形成最小“证据感”

### P2：无证据不强答

7. 在 question / followup / evaluation / report prompt 中加强“无证据不强答”约束
8. 对证据不足场景增加保守输出和默认话术

### P3：多角色建立在同一证据池之上

9. 让 panel 多角色共享同一批命中证据
10. 让 moderator 的选题、追问、评价都显式带上证据来源

### P4：报告证据化

11. 报告中补“本场训练主要依据 / 高频证据 / 建议训练方向”
12. 把高频短板和训练建议与轮次证据做更稳定关联

### P5：数据飞轮准备

13. 设计高质量样本沉淀格式
14. 建立幻觉案例、失败案例、优质追问案例归档机制
15. 建立离线评测样本集

### P6：任务型微调试点

16. 先只试点追问生成、结构化评分、报告总结等低风险任务
17. 严格要求离线评测达标后再考虑接入生产主链

### 当前执行顺序

本轮从 **P1-4 / P1-5 / P1-6** 开始，也就是：

- 先补切片命中原因
- 再补题目级 evidence trace
- 再把证据摘要接入报告

## 5. 短期最值得做的 RAG 升级

### 5.1 从“切片存在”升级为“证据可控”

当前切片能力已经存在，但还需要继续升级为真正的“证据块”：

- 每条切片都应能解释为什么命中
- 每轮输出都应能知道引用了哪些切片
- 需要把“证据不足”的情况显式表达出来

建议新增或强化：

- slice 命中原因记录
- question / followup / report 与 slice 的关联记录
- evidence trace 字段
- 命中置信度或规则评分摘要

### 5.2 建立“无证据不强答”的硬约束

当前系统已经有 prompt 结构，但还缺更明确的限制：

- 如果检索证据不足，就降低断言强度
- 无依据时允许给出“需要更多上下文”或“当前证据不足”
- 对岗位细节、项目细节、技术选型类问题，优先引用命中的切片或简历内容

这会直接降低：

- 幻觉
- 乱补细节
- 看似流畅但训练价值低的回答

### 5.3 每个阶段召回不同类型的证据

建议把不同阶段的检索进一步做清晰分工：

- `question`
  - 岗位核心要求
  - 高频技能点
  - 面试常见主题
- `followup`
  - 项目深挖点
  - 薄弱点相关证据
  - 上一轮回答暴露问题的补充切片
- `evaluation`
  - 标准答案要点
  - 评价维度
  - 关键扣分点 / 加分点
- `report`
  - 全局高频证据
  - 弱点聚合证据
  - 训练建议映射证据

### 5.4 把知识库从“文本补充”升级为“训练策略输入”

知识库不应只是塞给模型一段文本，而应在产品上承担更明确职责：

- 决定重点训练哪些能力
- 决定哪些题型更值得追问
- 决定评分更看重什么
- 决定报告输出更偏技术、项目、表达、业务哪一类建议

这意味着知识库不只是“背景资料”，而是：

**训练策略配置 + 证据来源**

### 5.5 增加“基于证据回答”的用户感知

短期不需要做复杂证据可视化，但可以逐步增强“证据感”：

- 报告中出现“本场训练主要依据以下能力点”
- 多视角结论更像“基于本轮表现得出”
- 用户感知到：

“AI 在基于我这场训练的证据说话”

## 6. 多角色能力的正确升级方向

当前多角色能力方向不需要推翻，但要明确规则：

### 6.1 多角色不是多说话，而是多视角判断

对用户可见的仍然是一个主持人。

多角色的作用是：

- 技术深挖
- 项目追问
- 业务场景
- 表达结构
- 压力质询

它们不是为了制造复杂感，而是为了让训练更接近真实面试的多维评估。

### 6.2 多角色必须基于同一证据池

下一阶段最关键的要求是：

**同一组证据在不同面试视角下被解释。**

例如：

- 技术视角看“方案是否合理”
- 项目视角看“细节是否可信”
- 表达视角看“回答是否结构化”
- 压力视角看“追问下是否稳定”

这样多角色不是各说各话，而是基于同一个证据池得出不同结论。

### 6.3 多角色的价值体现在三处

1. 更好的选题
2. 更有训练价值的追问
3. 更像真实面试反馈的综合评估

---

## 7. 微调建议：要做，但要改成“数据飞轮优先”

### 7.1 微调不应直接解决所有幻觉

短期不要把微调当成“幻觉修复总开关”。

更合理的定位是：

**用微调提升面试官风格一致性、追问质量稳定性、评分表达格式稳定性。**

### 7.2 微调最适合训练什么

真正适合微调的不是“岗位知识本身”，而是这些相对稳定的任务：

- 追问生成风格
- 结构化评分输出
- 报告总结格式
- 多角色 JSON 结构稳定性
- 训练建议表达一致性

### 7.3 微调数据从哪里来

当前项目已经很适合开始做数据飞轮：

- 用户真实面试记录
- 高质量追问样本
- 高质量评价样本
- 高质量报告样本
- 幻觉案例
- 不精确案例
- 被人工判定为“训练价值高”的对话片段

尤其要重点沉淀：

- 问题
- 追问
- 用户回答
- 评价结果
- 使用的切片证据
- 最终人工标签

### 7.4 微调前必须先做离线评测

在没有足够高质量样本前，不建议贸然把微调接进生产主链。

应该先建立：

- 黄金样本集
- 幻觉样本集
- 高质量追问样本集
- 报告质量样本集

然后对比：

- 当前主链路
- RAG 强化后链路
- 微调试点链路

### 7.5 微调上线方式建议

建议只做“小范围任务型试点”，例如：

- 只微调追问生成
- 只微调评分 JSON 格式稳定性
- 只微调报告总结表达

不要一上来做：

- 全链路主模型替换
- 面试全过程大一统微调
- 无离线评测的直接线上替换

## 8. 合并后的短期路线图

### Phase 0：稳定性与一致性收口

目标：

- 先把线上稳定性问题和部署一致性收口
- 让开发、测试、生产三套环境不再漂移

重点：

- migration 一致性
- schema / 代码一致性
- fallback 路径稳定
- 线上日志和调试信息可用

### Phase 1：RAG 证据化 1.0

目标：

- 让系统真正基于切片证据提问和评价

重点：

- slice 命中原因
- slice 引用链路
- 无证据降级
- Prompt 加强“无证据不强答”

### Phase 2：证据驱动的多角色协同

目标：

- 让 panel 真正建立在证据之上，而不是只做多角色 prompt 花活

重点：

- panel 角色共用同一证据池
- moderator 基于多视角和证据做选题 / 追问
- 失败时稳定 fallback 到单面试官模式

### Phase 3：报告与诊断证据化

目标：

- 让最终报告具备更强训练价值

重点：

- 分视角结论
- 高频短板聚合
- 训练建议
- 证据型总结

### Phase 4：微调数据飞轮

目标：

- 为微调做准备，而不是盲目上线微调

重点：

- 高质量样本沉淀
- 幻觉案例归档
- 评测集构建
- 失败样本分析

### Phase 5：任务型微调试点

目标：

- 在低风险、强格式任务上验证微调价值

重点：

- 追问生成
- JSON 稳定输出
- 报告总结风格
- 结构化评价表达

## 9. 具体落点建议

### 9.1 后端

建议优先继续落在这些已有主链文件上：

- `ai-interview-backend/app/services/client/interview_service.py`
- `ai-interview-backend/app/services/client/ai_service.py`
- `ai-interview-backend/app/services/client/position_knowledge_base_service.py`
- `ai-interview-backend/app/services/client/position_knowledge_base_slice_service.py`

重点不是新造体系，而是继续强化：

- evidence trace
- slice routing
- grounded prompt
- panel evidence sharing
- report signal aggregation

### 9.2 前端

前端仍坚持最小增强原则，不做大改版。

建议只持续补这几类感知：

- 当前模式提示
- 轻量多视角摘要
- 切片调试信息
- 报告中的训练建议与证据感

### 9.3 管理端

后台短期不做复杂运营平台，但可以逐步增强：

- 公共知识库质量控制
- 高质量切片管理
- 样本沉淀入口
- 失败案例查看入口

---

## 10. 短期不建议做的事

当前短期不建议优先投入：

- 一上来做全链路模型微调
- 频繁更换底层主模型
- 大规模前端交互改版
- 多窗口多 AI 聊天 UI
- 复杂向量数据库或重型 RAG 基础设施
- 大而全的知识库运营后台

这些事情不是永远不做，而是：

**现在做，性价比不高，还会分散主线。**

---

## 11. 建议的产品指标

为了让升级方向可验证，建议短期开始盯这几类指标：

- 幻觉率
- 无依据强答率
- 岗位贴合度
- 追问训练价值评分
- 报告可执行性评分
- slice 命中率
- 命中切片被实际引用的比例
- panel 模式与 single 模式的体验差异

如果后续要推进微调，这些指标也会成为评测基线。

---

## 12. 最终建议

这轮产品判断的核心结论是：

**短期不要把重点放在“换更强模型”，而要放在“让当前模型基于正确证据稳定工作”。**

所以最推荐的升级顺序是：

1. 先收口线上稳定性和一致性问题  
2. 再把 RAG 做成证据化、可解释、可追踪的主能力  
3. 让多角色协同建立在证据之上  
4. 让报告和训练建议真正具备证据感  
5. 同步沉淀数据飞轮，为后续小范围微调做准备  

如果这条路线执行好，`AI-Interview` 的产品形态会从：

**“会提问的 AI 面试官”**

走向：

**“能围绕岗位、简历、证据和训练目标做稳定训练闭环的 AI 面试训练系统”**
